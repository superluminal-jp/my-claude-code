# インタラクティブ・ドキュメントビルダー（対話型文書構築）

> `.claude/skills/minto-builder/SKILL.md` の日本語訳。原文（英語）が正本であり、本ファイルは参考訳。

## 目的

ユーザーが未整理のコミュニケーション課題を考え抜き、それを完成した文書へと仕上げる作業を支援する。

このスキルは協働的である。首尾一貫した文書がすでに存在することを前提としない。目的、読み手、中心的な主張、裏付け、根拠、行動を段階的に明確化してから、最終成果物を起草する。

内部では厳密な構造化原則を用いるが、ユーザーとの対話は平易な言葉で行う。

## このスキルを使う場面

以下のようなときに使用する。

- テーマはあるが、明確な文書がまだない
- 断片的なメモ、走り書き、あるいは競合するアイデアがある
- 対話を通じて論点を考え抜きたい
- 何を主メッセージにすべきか定まっていない
- 何を含め、何を省くかを決めかねている
- 対話を通じて、提案書・メモ・報告書・論文・プレゼンテーション・エグゼクティブサマリーを作り上げたい
- 執筆の前に考えを整理したいと明示的に求めている

ユーザーが提供しうるもの:

- 参考資料なし
- 断片的なメモ
- 部分的な根拠
- 複数の結論候補
- 未完成のアウトライン
- 直接の書き直しではなく、構想段階からの再構築が必要な草稿

## このスキルを使わない場面

ユーザーがすでに実質的な文書を持っており、主に以下を求めている場合は使用しない。

- 構造の診断
- そのままの最終版への書き直し
- 校正
- 文書構築を伴わない事実確認のみの調査

診断は `minto-reviewer` へ振り分ける。

直接の書き直しは `minto-rewriter` へ振り分ける。

## 対話の原則

対話は毎ターン進展させなければならない。

アシスタントの各ターンには、通常以下を含める。

1. 現時点で理解している内容の簡潔な要約
2. 最も重要な未解決の論点
3. 焦点を絞った質問1件、または密接に関連した質問のまとまり1件
4. 十分な材料が揃っていれば、有用な部分的成果物

長大なアンケートのような質問はしない。

すでに回答済みの質問を繰り返さない。

安全かつ後戻り可能な形で推測できる情報については尋ねない。

ユーザーから新しい情報が得られたら、次の質問をする前に理解内容を更新する。

## ユーザー向けの用語

ユーザーが明示的に求めない限り、フレームワーク固有の専門用語は使用しない。

避けるべき用語:

- SCQA
- 導入質問（Governing Question）
- 統括メッセージ（Governing Thought）
- 縦の論理（vertical logic）
- 横の論理（horizontal logic）
- 演繹的（deductive）
- 帰納的（inductive）
- MECE
- ピラミッド（pyramid）
- ノード（node）

代わりに平易な言葉を使う:

- 背景
- 何が変わったか
- 決めるべきこと
- 主なポイント
- 裏付けとなる理由
- 根拠
- リスク
- 行動
- 順序
- 不足している情報

## 内部状態の管理

以下の項目を内部的に保持し、必要なときのみ簡潔な形で提示する。

| 項目 | 意味 |
|---|---|
| 読み手 | 誰が読む・聞くのか |
| 期待する反応 | 読み手に何を決定・信頼・承認・実行してほしいか |
| 利用場面 | どこで、どのように文書が使われるか |
| 現在の理解 | 読み手がすでに知っている・受け入れていること |
| きっかけ | 何が変わり、なぜ今この文書が必要になったか |
| 中心的な問い | 文書が解決すべき決定事項・論点 |
| 主なポイント | 答え・推奨・結論となる主張 |
| 裏付けとなる論点 | 主な理由・発見・構成要素 |
| 根拠 | 事実・データ・事例・出典 |
| リスクと反論 | 読み手が提起しうる重大な異論 |
| 行動 | 決定事項・担当者・時期・次のステップ |
| 範囲 | 文書に含めること・含めないこと |
| 形式 | メモ・提案書・報告書・スライド・メール・論文などの種類 |
| 制約 | 分量・トーン・締め切り・必須内容・機密性 |

この表を毎ターン機械的に表示しない。

## 対話の段階

段階はあくまで内部的なものである。フレームワーク名でラベル付けしない。

### 1. アウトカムを定義する

以下を確定する。

- 想定する読み手
- 文書の種類
- 期待する反応
- 利用場面
- 締め切りや分量の制約（該当する場合）

後戻り可能な場合に限り、最も可能性の高い形式をデフォルトとする。

### 2. なぜこの文書が必要なのかを明らかにする

以下を明確化する。

- すでに事実であること
- 何が変わったか
- なぜ読み手が今それに注目すべきか
- そこから生じる決定事項・不確実性

抽象的な言明よりも具体的な事実を尋ねる。

### 3. 主なポイントを定める

ユーザーが一つの結論となる答えを選べるよう支援する。

複数の答えが有力な場合:

- 現実的な選択肢を2〜4件提示する
- それぞれのトレードオフを示す
- 根拠が許す限り一つを推奨する
- 読み手の権限や好みに依存する決定に限り、ユーザーに選択を求める

未解決の矛盾を隠さない。

### 4. 裏付けを構築する

選ばれた主なポイントについて、主要な裏付けグループを定める。

各グループは以下を満たさなければならない。

- 同種の問いに答えている
- 明確な境界を持つ
- 主なポイントに実質的に貢献する
- 抽象度の水準が一貫している
- 有用な順序に従っている

ユーザーが雑多なリストを提供した場合、以下に分類する。

- 理由
- 根拠
- 行動
- 事例
- リスク
- 背景

### 5. 完全性を検証する

以下を確認する。

- 不可欠な裏付けが欠けていないか
- 論点同士が重複していないか
- 主なポイントが手元の根拠より広すぎないか
- 想定される反論に対応できているか
- 該当する場合、行動に担当者・時期・依存関係があるか

文書の妥当性に最も影響する欠落項目のみを取り上げる。

### 6. 作業用の構造を示す

中心的な主張と主要な裏付けが安定したら、標準的なMarkdownの見出しまたは入れ子リストを用いて簡潔な構造を示す。

以下のような平易なラベルを使う。

- 主なポイント
- なぜ重要か
- 裏付けとなる理由
- 根拠
- リスク
- 推奨事項
- 次のステップ

文字ベースの図は使用しない。

実質的な選択肢についてのみユーザーに解決を求める。すべての見出しについて承認を求めない。

### 7. 有用な場合は段階的に起草する

長い文書の場合、協働の質が上がるならセクションごとに起草する。

各セクションの後に:

- 完成したセクションを示す
- 未解決の内容上の論点を一つ示す
- 焦点を絞った質問を一つする

短い文書の場合、構造が安定するまで待ってから成果物全体を起草する。

### 8. 仕上げる

以下がすべて満たされたら、完全な文書を作成する。

- 読み手と目的が明確である
- 主なポイントが一つに定まっている
- 主要な裏付けに一貫性がある
- 事実面の欠落が解消されているか、明示的に範囲設定されている
- 形式とトーンが判明している
- 妨げとなる矛盾が残っていない

最終文書には、明示的に求められない限り、対話履歴・内部フレームワーク・内部状態の表・構造分析を含めてはならない。

## 質問の方針

質問は、その答えが以下のいずれかを実質的に変える場合のみ行う。

- 中心的な主張
- 文書の範囲
- 必要な根拠
- 読み手の反応
- 文書の形式
- 影響の大きい推奨事項
- 捏造になりかねない主張

質問がブロッキングとなる場合は、以下の形式を用いる。

```markdown
**要決定事項:** [決定事項を一つ]

[質問]

デフォルト: [推奨されるデフォルト]  
代替案: [意味のある代替案]  
影響: [何が変わるか]
```

分割不可能な項目でない限り、1ターンにつき決定を要する質問は1件までとする。

## 部分的成果物の方針

早い段階から有用な進捗を示す。

部分的成果物の例:

- 一文で表した目的
- 主なポイントの候補
- 競合する二つの文書の方向性
- 整理された理由のリスト
- 作業中のセクション順序
- 冒頭部分の草稿
- 完成したセクション

ユーザーが容易に修正できる程度に、部分的成果物は簡潔にとどめる。

## 事実の整合性

以下を区別する。

- ユーザーから提供された事実
- 推論
- 仮定
- 提案された表現
- まだ必要な根拠

データ・引用・関係者の見解・日付・約束事項を捏造しない。

根拠が不足している場合は、以下のいずれかを行う。

- 主張の範囲を狭める
- 根拠を尋ねる
- 作業用構造の中で未解決として明示する
- 省略しても誤解を招かない場合は、最終文書から省く

## 最終成果物の契約

文書の準備が整ったら:

- 完成した成果物を主たる出力として返す
- 求められた文書の種類とトーンを用いる
- フレームワーク固有の用語を含めない
- 分析表を含めない
- ノードIDを含めない
- 求められない限り変更履歴を含めない
- 重大な未解決の仮定のみ、文書の外で開示する
- アスキーアート、罫線文字、テキスト図、文字ベースのツリー接続記号を使用しない

最終成果物は、それ以上の構造的な編集なしに使用できるものでなければならない。
