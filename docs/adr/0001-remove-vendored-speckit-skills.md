# 0001. Remove vendored speckit-* skills; Spec Kit stays per-project opt-in only

## Context and problem statement

This repository is a **user-level** Claude Code configuration: the entire
`.claude/` tree is designed to be copied wholesale to `~/.claude/`
(`install.sh`) so the same rules, skills, and hooks apply across every
project on the machine (README.md).

Until now it also vendored the 15 `speckit-*` skills generated by
`specify init` under `.claude/skills/speckit-*/`, and propagated them
further:

- `install.sh` copied the entire `skills/` directory — including
  `speckit-*` — into `~/.claude/skills`, and separately synced
  `.specify/extensions.yml` into a synthetic `~/.claude/.specify/`.
- `.claude/hooks/speckit-expand-update.sh` re-copied freshly regenerated
  `speckit-*` skills into `~/.claude/skills` after every `specify init
  --here --force` triggered by a `/speckit-*` invocation.

Upstream Spec Kit's own design is per-project: `specify init` scopes
`.specify/` and its generated command skills to the initializing project.
Treating the generated skills as shared, globally-installed configuration
duplicates Spec Kit's own update mechanism, produces copies that can drift
from the project that actually ran `specify init`, and does not match how
the tool is meant to be adopted (run `specify init` in each project that
wants it).

## Decision drivers

- Match Spec Kit's own per-project initialization model rather than
  fighting it with a parallel global-sync mechanism.
- Avoid stale/duplicated `speckit-*` skill copies drifting between this
  repo, other projects, and `~/.claude`.
- Keep this repository's own use of Spec Kit for its own development
  (`specs/001`–`010`, `.specify/`) working unchanged.

## Decision outcome

We will stop vendoring `speckit-*` skills and stop propagating any Spec
Kit state to the user-scope `~/.claude` install:

- Delete the 15 committed `.claude/skills/speckit-*/` directories and add
  `.claude/skills/speckit-*/` to `.gitignore`, so a local `specify init`
  regenerates them without re-entering version control.
- Remove the "sync regenerated skills to `~/.claude/skills`" step from
  `.claude/hooks/speckit-expand-update.sh`.
- Remove the "sync `.specify/extensions.yml` to `~/.claude/.specify`" step
  from `install.sh` (the git-extension auto-commit config step, which only
  edits this repo's own local `.specify/`, is unaffected).
- Update `README.md`, `README.ja.md`, `.claude/hooks/README.md`, and
  `.claude/rules/permissions.md` so none of them describe `speckit-*` as
  part of the bundled/shared config; the existing "Opt-in to spec-kit"
  section (run `specify init` per project) is now the only documented path.

`.specify/` and `specs/001`–`010` in this repo are untouched: they are this
project's own local Spec Kit installation and its SDD history. Continuing
spec-kit-based SDD here (or in any project) still works exactly the same
way — run `specify init` in that project.

### Consequences

- Positive: the distribution model matches Spec Kit's own per-project
  design; no more stale/duplicated `speckit-*` copies drifting between
  this repo and `~/.claude`; `install.sh`'s blanket `skills/` resync also
  retroactively removes any `speckit-*` directories a prior install left
  under `~/.claude/skills`.
- Negative: `/speckit-*` slash commands are no longer available globally
  out of the box after installing this config to `~/.claude` — any project
  that wants Spec Kit, including this one, must run `specify init` in it
  explicitly.

## Confirmation

- `tests/run-speckit-update.sh` asserts the hook never writes under
  `$HOME/.claude/skills`.
- `ls .claude/skills | grep speckit` returns nothing in this repository.
