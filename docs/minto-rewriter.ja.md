# エグゼクティブ構造リライター

> `.claude/skills/minto-rewriter/SKILL.md` の日本語訳。原文（英語）が正本であり、本ファイルは参考訳。

## 目的

既存の文書、または実質的な内容を持つ草稿を、完成し利用可能な最終文書へと変換する。

提供された範囲全体にわたって、厳密な構造化原則を内部的に適用する。分析上のフレームワーク、内部的なラベル、レビュー過程を出力に露出させない。

## このスキルを使う場面

ユーザーが以下のような既存のものを提供した場合に使用する。

- 文書の草稿
- 報告書
- メモ
- 提案書
- 記事
- 論文の一部
- エグゼクティブサマリー
- スライドのナラティブ
- 執筆に足る実質的な内容を含む詳細なアウトライン

典型的な依頼:

- これを書き直してほしい
- 文書を改善してほしい
- 論証を再構成してほしい
- エグゼクティブ向けに仕上げてほしい
- この草稿を最終版にしてほしい
- ロジックをより明確にしてほしい
- 洗練された版を作ってほしい
- 文書を簡潔・説得的・意思決定志向にしてほしい

## このスキルを使わない場面

ユーザーが主に以下を求めている場合は使用しない。

- 現在の構造を診断する
- 構造上の問題を説明する
- スコアや評価を受け取る
- 対話を通じて協働的に論証を見出す
- テーマのみで、実質的な資料がない状態から文書を作成する

構造診断は `minto-reviewer` へ振り分ける。

協働的な文書構築は `minto-builder` へ振り分ける。

## 出力の契約

完成した文書を主たる出力として返す。

明示的に求められない限り、以下を含めない。

- フレームワーク名
- 専門的な構造用語
- ノードID
- ステータスラベル
- 分析表
- 欠陥のリスト
- 書き直しの過程の説明
- 変更前後の比較
- 対応関係の一覧（ディスポジションマップ）
- 文書に先立つ独立したアウトライン
- なぜその構造が機能するかについてのメタ的な解説

読み手が、名前の付いたフレームワークから生成された文書としてではなく、自然に構造化された文書として結果を受け取れるようにする。

## 全面適用の要件

局所的な編集にとどめない。

提供された範囲全体を、以下を満たすように再構築する。

- 導入部が、文書の目的に必要な文脈だけを確立している
- 主な答え・推奨・結論となる主張が明示されている
- 各セクションが文書の中心的な目的を直接裏付けている
- 適切な場合、見出しが有用な結論を示している
- 関連する論点がまとめられている
- 異なる論理的種類が混在していない
- 各グループが一貫した順序に従っている
- 重要な重複と欠落が解消されている
- 根拠が、それが裏付ける主張のそばに配置されている
- 行動・理由・発見・リスク・事例が明確に分離されている
- 論証を追いやすくする移行（つなぎ）がある
- 結びが、決定事項・含意・次の行動を締めくくっている

原文を構造上の制約としてではなく、素材として扱う。

以下を行ってよい。

- 書き直す
- 順序を変える
- 統合する
- 分割する
- 移動する
- 圧縮する
- 裏付けられた含意のみを用いて拡張する
- 冗長または無関係な内容を削除する
- ラベルを情報量のある見出しに変換する
- 原文が裏付けない場合、中心的な結論を変更する

変更を最小限にとどめるためだけに、元の階層・文言・結論・順序を保持しない。

## 事実の整合性

以下を捏造しない。

- 事実
- 根拠
- 引用
- 数値指標
- 日付
- 出典
- 関係者の立場
- 因果関係の主張
- 約束事項

構造上必要な記述が裏付けられていない場合、最も影響の少ない有効な対処法を選ぶ。

1. 原文が裏付ける範囲まで主張を狭める
2. 制約があることを明確に述べる
3. 文書が明示的に作業中の草稿である場合、目に見えるプレースホルダーを使う
4. 省略しても誤解を招かない場合、裏付けのない主張を省く
5. 不足している情報が有効な最終文書の妨げとなる場合に限り、一つにまとめた確認質問を一つだけ行う

ユーザーが作業中の草稿を求めていない限り、公開可能な文書に内部用のプレースホルダーを挿入しない。

## 書き直しの手順

### 1. 執筆の前提を確立する

以下を推測または特定する。

- 読み手
- 目的
- 文書の種類
- 期待する決定・反応
- トーン
- 分量
- 必須のセクション
- 事実面の境界
- 利用場面

一つの未解決の決定が文書を実質的に左右する場合、一つにまとめた簡潔な質問を行う。それ以外は、明確に裏付けられる仮定のもとで進める。

### 2. 原文の内容を抽出する

以下を分離する。

- 中心的な主張
- 裏付けとなる主張
- 根拠
- 文脈
- リスク
- 行動
- 事例
- 制約
- 未解決の論点

これは内部的に行う。求められない限り、抽出結果を出力しない。

### 3. 文書の結論となる主張を定める

原文によって裏付けられ、読み手に適した中で最も強力な中心的主張を選ぶ。

原文に競合する結論が含まれる場合、ユーザーの明示した目的に最も合致するものを選ぶ。選択が真に妨げとなる場合は、一つの決定を求める。

### 4. 導入部を再構築する

以下を満たす導入部を作成する。

- 関連する共有された文脈から始める
- なぜ今この文書が重要なのか、その理由を提示する
- 決定事項または論点を特定する
- 文書の中心的な答えを早い段階で述べる

これを文書の種類に合わせて調整する。媒体が直接的な導入を必要とする場合、形式的な導入を無理に用いない。

### 5. 本文を再構築する

完全な階層構造を構築する。

各セクションについて:

- 一つの主なポイントを述べる
- そのポイントを裏付ける内容のみを含める
- 比較可能な項目をまとめる
- 一つの順序原則を用いる
- 根拠を、それが裏付ける主張の近くに配置する
- 分析と行動を分離する
- 含意を明確にする

### 6. 結びを再構築する

文書に適した形式で締めくくる。

- 推奨事項
- 決定の要請
- 含意
- 行動計画
- 結論
- 次のステップ

文書がすでに明確に締めくくられている場合、汎用的なまとめを付け加えない。

### 7. 編集・校正する

以下を確認する。

- 事実面の忠実性
- 内部的な整合性
- 用語の一貫性
- 読み手に適した深さ
- 段落の焦点
- 能動態
- 不要な繰り返し
- 裏付けのない断定
- 書式
- 文法・句読点

### 8. 完成した成果物のみを返す

すべての実質的な最終文章を、適切な完成形の出力形式に収める。

重大な制約・未解決の事実面のギャップ・仮定を開示する必要がある場合に限り、文書の外に短い注記を加える。

## 形式のルール

想定される体裁を保持または適応させる。

- メールはメールのまま
- メモはメモのまま
- 報告書は報告書のまま
- スライドのストーリーラインは、スライドタイトルと裏付け内容のまま
- 学術的な文章は、適切な限定・留保を保ったまま
- エグゼクティブ向けの文章は、決定事項または含意から始める

すべての成果物を汎用的な報告書に変換しない。

## ユーザー向けの言葉

ユーザーが明示的に構造分析を求めない限り、以下は使用しない。

- SCQA
- 導入質問（Governing Question）
- 統括メッセージ（Governing Thought）
- 縦の論理（vertical logic）
- 横の論理（horizontal logic）
- 演繹的グルーピング（deductive grouping）
- 帰納的グルーピング（inductive grouping）
- MECE
- ピラミッド（pyramid）
- ノード（node）
- 構造上のステータスラベル

通常の、その領域に適した言葉を使う。

## 品質ガードレール

- アウトラインや修正リストではなく、完成した文書を返す。
- 提供された範囲全体にわたって構造化原則を適用する。
- 内部の推論過程やフレームワーク用語を露出させない。
- 事実や根拠を捏造しない。
- 文体上の都合で、裏付けのない主張を残さない。
- 完成した文書の前に分析を含めない。
- アスキーアート、罫線文字、テキスト図、文字ベースのツリー接続記号を使用しない。
- 通常の見出し、段落、リストを使用し、表は文書に適切な場合のみ使用する。
