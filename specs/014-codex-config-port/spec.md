# Feature Specification: `.claude` 設定の Codex CLI への包括移植と挙動同期

**Feature Branch**: `014-codex-config-port`

**Created**: 2026-07-20

**Status**: Draft

**Input**: User description: "@.claude の設定を codex でも再現して同様に使えるように移植を行う。https://learn.chatgpt.com/docs　https://code.claude.com/docs/en/overview　codex, claude code それぞれの公式ドキュメントを参照して最適な方式で挙動を同期させる"

**Clarification (2026-07-20, maintainer)**: `.claude` と同様に、`.codex` と `.agents` は**ユーザールートディレクトリ(`~/`)に置かれるユーザー設定**とする前提で設計する。つまりリポジトリ内の `.codex/`・`.agents/` は編集・版管理用のソースであり、`.claude/` → `~/.claude/` と同じモデルで `~/.codex/`・`~/.agents/` へ展開されて初めて有効になる。

## 概要

このリポジトリの `.claude/` 設定(指針プロース、ルール、スキル、フック、権限設定、MCP カタログ、カスタムコマンド)を、OpenAI Codex CLI でも**同等の挙動**で使えるように移植を完成させ、以後両ツールの挙動が乖離しない仕組みを設ける。展開モデルはユーザースコープを主とする: リポジトリで版管理する `.codex/`・`.agents/` を、既存の `.claude/` → `~/.claude/` ミラーと同じ方式で `~/.codex/`・`~/.agents/` に展開し、Codex CLI にはユーザー設定として読ませる。

先行フィーチャーとの関係:

- `specs/012-cross-agent-guardrail-migration/` — 14 項目のフック/ルールについて「共通化するか・どの強度で」の判断を記録済み(決定レコードは確定、再審理しない)。
- `specs/013-cross-agent-guardrail-implementation/` — 012 の判断のうちガードレール中心の部分を実装済み: `AGENTS.md`、`.codex/hooks/` の 3 アダプタ(破壊的コマンド/pre-edit/post-edit)、`.agents/skills/` へのスキル公開(手書きスキルはシンボリックリンク、`speckit-*` は既存フックによる同期)。

本フィーチャーは **013 が扱わなかった残りの構成要素**(権限設定、MCP 接続カタログ、カスタムコマンド、ユーザースコープ展開)と、**既移植分の劣化修復・恒常同期**を扱う。実際、013 実装後のルール整理(コミット 33c82eb)により `.agents/skills/` に壊れたシンボリックリンクが 3 件(`advisor`, `domain-model`, `ubiquitous-language` — 参照先が `.claude/skills/` から削除済み)発生しており、「一度移植して終わり」では同期が保てないことが実証されている。また作業ツリーには未確定の再配置(ルート `AGENTS.md` の中身を `.codex/AGENTS.md` へ移す変更)が残っており、Codex CLI が公式にどの位置の `AGENTS.md` を読むかに照らした確定が必要である。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 構成要素の完全対応表と移植完了 (Priority: P1)

メンテナが `.claude/` 配下の全構成要素について「Codex ではどこで・どう同じ挙動になるか(または、ならない理由)」を 1 つの対応表で確認でき、対応表上「移植対象」とされた要素がすべて実際に Codex 側に存在する。

**Why this priority**: 「同様に使える」の到達判定は網羅的な対応表なしには検証不能。抜け漏れ(今回発見された壊れリンクのような)を防ぐ土台であり、他のすべてのストーリーの受け入れ判定の基準になる。

**Independent Test**: 対応表を開き、`.claude/` 配下の実ファイルと突き合わせて、分類されていない要素が 0 件であること、「移植済み」項目の Codex 側実体が存在することを確認する。

**Acceptance Scenarios**:

1. **Given** `.claude/` 配下の全構成要素(CLAUDE.md・rules・skills・hooks・settings の permissions/hooks/その他・commands・`.mcp.json`)、**When** 対応表と突き合わせる、**Then** 全要素が「移植済み / 本機能で移植 / 対象外(理由付き)」のいずれかに分類されている
2. **Given** 対応表で「移植済み」「本機能で移植」とされた要素、**When** Codex 側の対応物(ファイル・リンク・設定)を確認する、**Then** すべて実在し、壊れた参照が 0 件である
3. **Given** 対象外とされた要素(例: statusline)、**When** 対応表を読む、**Then** Codex に対応機構がない等の理由が明記されている

---

### User Story 2 - 指針プロースが Codex に確実に読まれる (Priority: P1)

Codex CLI のセッションを開始すると、Claude Code の `CLAUDE.md` + `.claude/rules/` に相当する指針(ツール使用原則、MCP 使い分け、編集規約、Spec Kit 運用など)が、Codex が公式に読み込む位置・形式の `AGENTS.md` から自動的に適用される。

**Why this priority**: 現在の作業ツリーはルート `AGENTS.md` を実質空にして `.codex/AGENTS.md` に内容を移した未確定状態にある。確定した前提(`.codex` はユーザー設定のソースで `~/.codex/` へ展開)では、リポジトリの `.codex/AGENTS.md` は展開されて `~/.codex/AGENTS.md`(Codex のグローバル指針位置)として読まれる設計になるが、**展開されない限り Codex はリポジトリ内の `.codex/AGENTS.md` を読まない**。公式の探索階層(グローバル→プロジェクトルート→カレント方向、サイズ予算 32 KiB)に照らして「どの内容がユーザースコープ(全プロジェクト共通)で、どの内容がプロジェクトルート `AGENTS.md` に残るべきか」を確定させないと、指針全体が黙って無効化されうるため最優先。

**Independent Test**: Codex CLI セッションを対象リポジトリで起動し、指針に含まれる固有の指示(例: AWS 質問時の MCP サーバ選択)への応答挙動、またはセッションが読み込んだプロジェクトドキュメントの確認手段で、指針が実際に読まれていることを確認する。

**Acceptance Scenarios**:

1. **Given** ユーザースコープ展開(`~/.codex/` 等)を実施済みの環境でリポジトリを開いた Codex CLI セッション、**When** セッションが読み込む指針を確認する、**Then** 移植した指針プロースが公式探索位置(ユーザースコープのグローバル指針、および必要ならプロジェクトルート `AGENTS.md`)から読み込まれている(サイズ予算内)
2. **Given** 未確定の再配置(ルート `AGENTS.md` 空化 + `.codex/AGENTS.md`)、**When** 「`.codex` はユーザー設定のソース」という前提と公式探索仕様に突き合わせる、**Then** ユーザースコープに置く内容とプロジェクトルートに残す内容の切り分けが確定し、展開されず読まれもしない位置に指針が残っていない
3. **Given** Claude Code セッション、**When** 同じリポジトリで作業する、**Then** 従来どおり `CLAUDE.md`/`.claude/rules/` の指針が適用され、移植による劣化がない

---

### User Story 3 - 権限・承認挙動の同等再現 (Priority: P2)

Codex CLI で作業するユーザが、Claude Code の `settings.json` permissions(allow / ask / deny)とおおむね同じ体験を得る: 破壊的操作は確認または拒否され、リポジトリの検証コマンド類は確認なしで通る。

**Why this priority**: ガードレールの中核(破壊的コマンド遮断)は 013 で移植済みだが、permissions の allow リスト(検証スイートやリンタを確認なしで実行)と ask/deny の細目は未移植であり、Codex 側だけ確認プロンプト過多または保護不足になる。

**Independent Test**: 代表操作サンプル(許可されるべき検証コマンド、確認されるべき git 書き込み、拒否されるべき credential 読み取り)を両ツールで試行し、判定(通過/確認/拒否)が一致することを確認する。

**Acceptance Scenarios**:

1. **Given** Codex CLI セッション、**When** 破壊的コマンド(force push、`rm -rf` 等)を実行しようとする、**Then** Claude Code と同様にブロックまたは確認が入る(013 実装の維持)
2. **Given** Codex CLI セッション、**When** credential パス(`.env`、`secrets/` 等)の読み取り・書き込みを試みる、**Then** 拒否される
3. **Given** 完全一致が不可能な権限項目(両ツールの権限機構の表現力差)、**When** 対応表を読む、**Then** 差分と受容理由が制約として明記されている

---

### User Story 4 - MCP サーバが Codex でも同じカタログで使える (Priority: P2)

Codex CLI ユーザが、`.mcp.json` に定義された 6 つの MCP サーバ(aws-knowledge、aws-documentation、bedrock-agentcore、strands-agents、google-developer-knowledge、microsoft-learn)と同じカタログを Codex から利用でき、AWS/GCP/Azure 質問時の使い分けルールも適用される。

**Why this priority**: MCP 使い分けは指針プロースの中で「必須」とされている運用ルールであり、接続がなければ指針が実行不能になる。ただし接続形式は両ツールで非互換(013 で確認済み)のため、単一ファイル統一ではなく同等カタログの提供になる。

**Independent Test**: Codex CLI セッションで 6 サーバが接続済みとして列挙され、代表クエリ(例: AWS ドキュメント検索)が成功することを確認する。

**Acceptance Scenarios**:

1. **Given** Codex CLI セッション、**When** 利用可能な MCP サーバを列挙する、**Then** `.mcp.json` と同じ 6 サーバが使える
2. **Given** 認証キーが必要なサーバ(google-developer-knowledge)、**When** 接続設定を確認する、**Then** 秘密情報は設定ファイルに直接書かれておらず、環境変数等で注入される
3. **Given** `.mcp.json` にサーバが追加・削除された、**When** 同期手順(または検証)を実行する、**Then** Codex 側カタログとの乖離が検出・解消される

---

### User Story 5 - スキルとカスタムコマンドの単一ソース利用 (Priority: P2)

Codex CLI ユーザが、Claude Code と同じスキル群(手書きスキル + `speckit-*`)を Codex のスキル発見機構から起動でき、`/verify-config` に相当する検証入口も Codex から呼べる。スキル本体は単一ソース(`.claude/skills/`)のまま複製されない。

**Why this priority**: スキル公開は 013 で実装済みだが、現に壊れリンク 3 件という劣化が起きており、修復と「参照先が消えたら検出される」保証が必要。カスタムコマンド(現状 `verify-config` のみ)は未移植。

**Independent Test**: Codex CLI セッションでスキル一覧に全対象スキルが表示され、壊れたリンクが 0 件で、検証入口の呼び出しが Claude Code の `/verify-config` と同じチェックを実行することを確認する。

**Acceptance Scenarios**:

1. **Given** `.agents/skills/`(リポジトリ内ソース)と展開後の `~/.agents/skills/`、**When** リンク健全性を検証する、**Then** どちらでも壊れたシンボリックリンクが 0 件である(現状の 3 件 — 参照先スキルは `~/.claude/skills/` には存在する — は、ユーザースコープ前提での単一ソースの取り方を対応表で分類した上で解消する)
2. **Given** Codex CLI セッション、**When** スキルを列挙・起動する、**Then** 対応表で「移植対象」とされた全スキルが発見・起動できる
3. **Given** Codex CLI セッション、**When** 設定検証の入口を呼ぶ、**Then** `/verify-config` と同等の検証が実行できる

---

### User Story 6 - ドリフト検知による恒常同期 (Priority: P3)

メンテナが `.claude/` 側または Codex 側のどちらかだけを変更してしまった場合、リポジトリの検証スイートがその乖離(壊れリンク、対応表との不整合、MCP カタログ差分、共有スクリプト参照切れ)を検出し、修正を促す。

**Why this priority**: 単一ソース化できない部分(プロースの二重管理、MCP 接続の二形式)は人手同期に頼ると必ず乖離する — 壊れリンク 3 件がその実証。ただし P1〜P2 の移植が完了して初めて意味を持つため P3。

**Independent Test**: 対応ペアの一方を意図的に変更(例: `.claude/skills/` からスキルを削除、`.mcp.json` にサーバを追加)し、検証スイートが失敗して乖離箇所を指摘することを確認する。

**Acceptance Scenarios**:

1. **Given** `.claude/skills/` からスキルディレクトリを削除した状態、**When** 検証を実行する、**Then** `.agents/skills/` 側の壊れリンクが検出される
2. **Given** `.mcp.json` と Codex 側接続カタログのサーバ集合が異なる状態、**When** 検証を実行する、**Then** 差分が検出される
3. **Given** すべて同期された状態、**When** 検証を実行する、**Then** すべてのチェックが通過する

---

### Edge Cases

- 指針プロースが将来肥大化して Codex の 32 KiB 既定予算(`project_doc_max_bytes`)を超えた場合、超過分が黙って切り捨てられる/読まれない恐れ — 検証でサイズを監視する。
- Codex がプロジェクトの `AGENTS.md` を読む位置は公式探索階層で決まる。`.codex/AGENTS.md` のような非公式位置に置くと黙って無視される — 配置確定は公式ドキュメントの記述に基づき、実挙動でも確認する。
- シンボリックリンクを扱えない環境(一部の Windows 構成、シンボリックリンク非対応のチェックアウト)ではスキル公開が壊れる — 検証がそれを検出し、対応表が制約として記録する。
- Codex のネイティブ権限機構(接頭辞ベースの許可/確認/拒否ルール)は、Claude Code 側フックの正規表現判定(例: `&&` 連結後方の `rm -rf` 検出)より表現力が低い — 完全一致できない判定は「共有スクリプト経由のフック」側でカバーし、残差は対応表に明記する。
- MCP サーバの認証キー(環境変数)が未設定の環境では該当サーバのみ接続失敗する — 他サーバと指針の適用は影響を受けないこと。
- `speckit-*` スキルの再同期(既存の自動更新フロー)が `.agents/skills/` を再生成する際、手書きスキルのリンクを消さない/上書きしないこと。
- ユーザースコープ展開(`~/.codex` 側)とプロジェクトスコープ(リポジトリ内)で同名設定が競合した場合、Codex の公式優先順位に従い、期待と異なる上書きが起きないことを対応表に記録する。
- リポジトリ内の相対シンボリックリンク(例: `.agents/skills/adr -> ../../.claude/skills/adr`)は、`~/.agents/` へ展開すると参照先が変わり壊れうる — 展開方式は展開先での解決可能性(例: `~/.claude/skills/` を指す)を保証する。
- Codex がユーザースコープでスキルを走査する公式位置が `~/.agents/skills/` と異なる場合(公式ドキュメントの記載が前提と食い違う場合)は、前提を機械的に押し通さず、公式位置に合わせた展開先を採用して対応表に記録する。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: `.claude/` 配下の全構成要素(`CLAUDE.md`、`rules/*.md`、`skills/*`、`hooks/*`、`settings.json` の permissions・hooks・その他キー、`commands/*`、および `.mcp.json`)を列挙し、各要素を「移植済み(013 まで)/ 本機能で移植 / 対象外(理由付き)」に分類した対応表を、移植物の近傍に配置された文書として提供しなければならない。分類率は 100% とする。
- **FR-002**: 対応表の各「移植」項目には、Claude Code 側ソース、Codex 側実体、同期方式(単一ソース参照 / 自動同期 / 検証付き二重管理)、および挙動差分(あれば)を記載しなければならない。
- **FR-003**: 指針プロース(`CLAUDE.md` + `rules/` のうち 012 で共通化と決定済みの内容)は、Codex CLI が公式探索階層で確実に読み込む位置の `AGENTS.md` に、既定サイズ予算(32 KiB)内で配置しなければならない。配置は確定前提に従う: ユーザースコープ(全プロジェクト共通)の内容はリポジトリの `.codex/` をソースとして `~/.codex/` へ展開し、プロジェクト固有の内容(があれば)はプロジェクトルートの `AGENTS.md` に置く。作業ツリーに残る未確定の再配置はこの切り分けに従って確定させ、展開されず読まれもしない位置に指針を残さない。
- **FR-004**: 012/013 で確定済みの判断(14 項目の共通化強度、ガードレールの共有スクリプト+アダプタ構成、Cursor 対象外)は再審理せず、本機能はその上に積み増す。既存の 3 アダプタ(破壊的コマンド / pre-edit / post-edit)の挙動は維持し、検証対象に含める。
- **FR-005**: `settings.json` の permissions(allow / ask / deny)に相当する権限挙動を、Codex のネイティブ権限機構で再現しなければならない。少なくとも: (a) リポジトリの検証・リント系コマンドは確認なしで実行できる、(b) credential パスの読み書きは拒否される、(c) 破壊的 git 操作は確認または拒否される。ネイティブ機構で表現できない判定は既存の共有スクリプト+フック側の担保とし、残差は対応表に制約として記録する。
- **FR-006**: `.mcp.json` に定義された全 MCP サーバと同一のカタログを、Codex CLI が読む接続設定として提供しなければならない。秘密情報(API キー等)を接続設定ファイルに平文で含めてはならない。
- **FR-007**: 対応表で移植対象とされた全スキルが Codex のスキル発見機構から起動できなければならない。スキル本体は `.claude/skills/` を単一ソースとし、複製を作らない。現存する壊れたシンボリックリンク 3 件(`advisor`、`domain-model`、`ubiquitous-language`)は、参照先の扱いを対応表で分類した上で解消する。
- **FR-008**: `/verify-config` に相当する設定検証の入口を、Codex のカスタムコマンド機構(プロンプト)として提供しなければならない。検証ロジック自体は両ツールで共有し、二重実装しない。
- **FR-009**: リポジトリの検証スイートに、両ツール間の同期健全性チェックを追加しなければならない。少なくとも: 壊れたシンボリックリンクの検出、MCP カタログの集合一致、指針プロースのサイズ予算内チェック、共有スクリプト参照の解決可能性。乖離時は検証が失敗し、乖離箇所を特定できる出力を返す。
- **FR-010**: `.claude/` → `~/.claude/` のユーザースコープ展開(既存 install.sh)と同じモデルで、リポジトリの `.codex/` → `~/.codex/`、`.agents/` → `~/.agents/` の展開を提供しなければならない(確定前提)。展開後の `~/.agents/`・`~/.codex/` 内の参照(スキルの単一ソース参照等)は、展開先でも解決可能でなければならない(リポジトリ相対の参照が `~` 配下で壊れないこと)。**スキルはディレクトリの単純ミラーではなく、展開先(`~/.claude/skills/<name>`)を指すリンクとして個別に再生成する**(Edge Cases 参照)。
- **FR-011**: Codex に対応機構が存在しない要素(例: statusline、Claude Memory 連携)は移植せず、対象外理由と代替(プロース注意書き等、012 の決定に従う)を対応表に記録しなければならない。なお、プロンプト内秘密情報検知(user-prompt-submit)は当初この分類を想定していたが、planning 時の公式ドキュメント調査で Codex hooks に同等イベント(UserPromptSubmit)の存在が確認されたため(research.md R4)、013 の前例に倣い「共有スクリプト+アダプタ」での移植対象に昇格する。012 Q5 の記録は書き換えず、昇格根拠は本 spec と research.md に残す。
- **FR-012**: 本機能で変更・追加されるすべての文書(対応表、AGENTS.md、hooks README 等)は、変更と同一のコミット単位で更新され、実挙動と乖離した記述を含んではならない。

### Key Entities

- **構成要素対応表**: `.claude/` の各構成要素と Codex 側対応物のマッピング。属性: Claude 側ソースパス、Codex 側実体、分類(移植済み/本機能で移植/対象外)、同期方式、挙動差分・制約、対象外理由。
- **指針プロース(AGENTS.md)**: 両ツール共通の作業指針。Claude Code 側は `CLAUDE.md`+`rules/`、Codex 側は公式探索位置の `AGENTS.md`。サイズ予算と探索階層の制約を持つ。
- **スキル**: `.claude/skills/<name>/SKILL.md` を単一ソースとする再利用可能な作業手順。Codex 側は発見機構向けの公開エントリ(リンク/同期)として現れる。
- **ガードレールフック**: 破壊的コマンド遮断・編集保護・編集後整形の共有判定ロジックと、ツール別アダプタ(013 実装済み)。
- **権限ポリシー**: 操作を許可/確認/拒否に振り分ける規則集合。Claude Code 側は settings.json permissions、Codex 側はネイティブ権限機構+フック。
- **MCP 接続カタログ**: 6 サーバの接続定義。両ツールで形式非互換のため、集合一致を検証で担保する対応ペア。
- **同期健全性チェック**: 対応ペアの乖離を検出する検証群。検証スイートの一部。

## Success Criteria *(mandatory)*
