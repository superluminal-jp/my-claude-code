# Feature Specification: Automatic Skill Routing

**Feature Branch**: `001-skill-auto-routing`  
**Created**: 2026-05-06  
**Status**: Draft  
**Input**: User description: "コードを書くときは必ず coder スキルを使用するように。ドキュメントやスライドを作成するときは必ず editor スキルを使用するように。その挙動が可能であれば rules の記述は最小限で良いはず。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Code Request Auto-Routes to Coder (Priority: P1)

ユーザーがコードの実装・修正・リファクタリング・テスト追加を依頼したとき、Claude は明示的な指示なしに `coder` スキルを自動的にロードして作業を開始する。

**Why this priority**: コード作業はこのプロジェクトで最も頻繁に行われるタスクであり、毎回手動でスキルを指定する手間を排除することで一貫した品質が確保できる。

**Independent Test**: ユーザーが "バグを修正して" と入力するだけで、`coder` スキルがロードされ、コード変更が行われることを確認できる。

**Acceptance Scenarios**:

1. **Given** ユーザーがコードの実装・修正・リファクタリングを依頼する、**When** Claude がリクエストを受け取る、**Then** `coder` スキルが自動的にロードされ、コード変更が行われる
2. **Given** ユーザーがテストの追加を依頼する、**When** Claude がリクエストを受け取る、**Then** `coder` スキルが自動的にロードされる
3. **Given** ユーザーがデバッグを依頼する、**When** Claude がリクエストを受け取る、**Then** `coder` スキルが自動的にロードされる

---

### User Story 2 - Document Request Auto-Routes to Editor (Priority: P2)

ユーザーがドキュメント・スライド・チャート・翻訳・文章編集を依頼したとき、Claude は明示的な指示なしに `editor` スキルを自動的にロードして作業を開始する。

**Why this priority**: ドキュメント作成もコード作業に次いで頻繁に行われるタスクであり、適切なスキルが自動適用されることで成果物の品質が向上する。

**Independent Test**: ユーザーが "READMEを書いて" と入力するだけで、`editor` スキルがロードされ、ドキュメントが作成されることを確認できる。

**Acceptance Scenarios**:

1. **Given** ユーザーがドキュメント・スライドの作成を依頼する、**When** Claude がリクエストを受け取る、**Then** `editor` スキルが自動的にロードされる
2. **Given** ユーザーが文章の翻訳・編集を依頼する、**When** Claude がリクエストを受け取る、**Then** `editor` スキルが自動的にロードされる
3. **Given** ユーザーがチャートやスプレッドシートの作成を依頼する、**When** Claude がリクエストを受け取る、**Then** `editor` スキルが自動的にロードされる

---

### User Story 3 - Minimal Rules Description (Priority: P3)

自動ルーティングが確実に機能する場合、`CLAUDE.md` および `rules/skill-routing.md` のスキル説明文が最小限に削減される。

**Why this priority**: ルールの冗長な記述はメンテナンスコストを増加させる。自動ルーティングが機能していれば詳細な説明は不要になる。

**Independent Test**: 削減後のルールで、コードリクエストに `coder`、ドキュメントリクエストに `editor` が正しく適用されることを確認できる。

**Acceptance Scenarios**:

1. **Given** 自動ルーティングが正常に機能している、**When** ルール記述を精査する、**Then** ルーティング条件の重複・冗長な説明が除去されている
2. **Given** 削減されたルールが適用されている、**When** コード・ドキュメントの各リクエストを行う、**Then** 正しいスキルが変わらずロードされる

---

### Edge Cases

- コードとドキュメントの両方が混在するリクエスト（例: "実装してREADMEも書いて"）の場合: `coder` → `editor` の順に両スキルをロードして処理する（FR-004）
- `clarifier` スキルは軽微な曖昧さでもトリガーされる。不明確なまま作業を進めない（FR-005）
- spec-kit のスラッシュコマンド（`/speckit-*`）は自動ルーティングの対象外（FR-006）

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: コードの実装・修正・リファクタリング・テスト・デバッグを含むリクエストが来たとき、`coder` スキルを自動的にロードしなければならない
- **FR-002**: ドキュメント・スライド・チャート・翻訳・文章編集を含むリクエストが来たとき、`editor` スキルを自動的にロードしなければならない
- **FR-003**: 自動ルーティングが正確に機能する場合、`CLAUDE.md` および `rules/skill-routing.md` のスキル記述はトリガー条件のみ（各スキル 1 行）に削減し、説明文・例は除去する
- **FR-004**: コードとドキュメントが混在するリクエストでは、`coder` → `editor` の順に両スキルを順番にロードして各タスクを処理しなければならない
- **FR-005**: `clarifier` スキルは自動ルーティングの対象に含め、`rules/clarify.md` の判定基準（intent/scope/acceptance/constraint gap、セキュリティ・コンプライアンスリスクを含む）のいずれかに該当する場合、軽微な曖昧さであっても自動適用されなければならない。不明確なまま作業を進めることを防ぎ、ユーザーの要件理解を促進する。
- **FR-006**: spec-kit スラッシュコマンド（`/speckit-*`）は独自のプレイブックを持つため、自動ルーティングの対象外とする
- **FR-007**: ルールの検証用プロンプトシナリオを `tests/` フォルダに蓄積し、コードに対する TDD と同様にプロンプトに対するテストを実行できる環境を整備しなければならない。各シナリオはリクエストパターン・期待スキル・期待動作を記述する。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: コード関連リクエストの 100% で `coder` スキルが自動ロードされ、ユーザーが明示的にスキルを指定する必要がない
- **SC-002**: ドキュメント関連リクエストの 100% で `editor` スキルが自動ロードされ、ユーザーが明示的にスキルを指定する必要がない
- **SC-003**: ルール削減後の精度検証は `tests/` フォルダに蓄積したプロンプトシナリオ（コード・文書・混在・曖昧の各カテゴリを網羅）を用いて実施し、全シナリオで期待スキルがロードされることを確認できる
- **SC-004**: `CLAUDE.md` および `rules/skill-routing.md` のスキルルーティング記述がトリガー条件のみ（各スキル 1 行）に収まる

## Clarifications

### Session 2026-05-06

- Q: コードとドキュメントが混在するリクエストでのスキルロード順序は？ → A: `coder` → `editor` の順に両スキルを順番にロード
- Q: ルール削減後に残すスキル記述の粒度は？ → A: トリガー条件のみ（各スキル 1 行、説明・例なし）
- Q: `clarifier` の自動ロード条件は？ → A: `rules/clarify.md` 全基準（セキュリティ・コンプライアンスリスク含む）、軽微な曖昧さでもトリガー（不明確なまま進めない、要件理解促進を優先）
- Q: ルール削減後の精度検証方法は？ → A: `tests/` フォルダにプロンプトシナリオを蓄積し TDD 的に運用（各シナリオ: リクエスト・期待スキル・期待動作を記述）

## Assumptions

- 自動ルーティングは `CLAUDE.md` の指示を通じて Claude の応答プロセスで実現可能である
- `coder`・`editor`・`clarifier` の 3 スキルのルーティングが対象であり、他のスキルは現状維持
- spec-kit スラッシュコマンドは自動ルーティングの対象外（各コマンドが独自プレイブックを持つため）
- ルール削減はルーティング精度の検証後に実施する（削減 → テスト → 調整のサイクル）
- `tests/` フォルダはプロンプトシナリオの蓄積場所として新規作成する（既存の `specs/` とは独立）
