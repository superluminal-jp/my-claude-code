# Feature Specification: DDD ドメインモデル管理スキル

**Feature Branch**: `005-ddd-domain-model-skill`  
**Created**: 2026-05-09  
**Status**: Draft  
**Input**: User description: "Eric Evan's DDD に則って、ベストプラクティスに従ってドメインモデルを作成するスキルを作成する"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 会話からドメイン語彙を受動収集する (Priority: P1)

開発者またはアーキテクトが通常の会話の中でドメイン語彙（集約・エンティティ・値オブジェクト・ドメインイベント・不変条件）を含む発言をすると、スキルがそれを検出してキューに積む。会話フローを中断せず、次のターンで新規候補がなくなったときに候補を提示する。

**Why this priority**: モデルの素材となる語彙を手動入力なしに収集できることが、このスキルの核心価値である。

**Independent Test**: ユーザーが「注文は複数の明細を持つ」と発言したあと、スキルが「注文（Order）を集約候補としてキューに追加しました」と返せば、この Story は独立して検証可能。

**Acceptance Scenarios**:

1. **Given** ユーザーが「注文は複数の明細を持つ」と発言した, **When** スキルがターンを処理した, **Then** 「注文」が集約候補としてキューに追加され、会話が中断されない
2. **Given** 前のターンで新規候補がなかった, **When** スキルが次のターンを処理した, **Then** キューに積まれた候補一覧を提示して確認を求める
3. **Given** 「〇〇ID」「〇〇番号」という語が登場した, **When** スキルが処理した, **Then** エンティティ候補としてキューに追加される
4. **Given** 「注文が確定された」という過去形・受動形表現があった, **When** スキルが処理した, **Then** ドメインイベント候補としてキューに追加される

---

### User Story 2 - 明示コマンドでモデルファイルを生成・更新する (Priority: P2)

開発者がスキルを明示的に呼び出し、指定した境界づけられたコンテキスト（Bounded Context）のモデルファイルを新規作成または更新する。スキルは差分を提示し、ユーザーが確認してからファイルに書き込む。

**Why this priority**: 受動収集した語彙をファイルに永続化する手段がなければ収集は無意味になる。

**Independent Test**: 「注文コンテキストのモデルを作成して」と指示したあと、スキルが差分を提示し、確認後に `docs/models/order.md` が生成されれば独立検証可能。

**Acceptance Scenarios**:

1. **Given** コンテキスト「注文」を指定してスキルを呼び出した, **When** スキルが応答した, **Then** Mermaid classDiagram・集約テーブル・エンティティテーブル・VO テーブル・ドメインイベントテーブル・不変条件リストを含む差分が提示される
2. **Given** 差分を提示されたユーザーが確認した, **When** スキルが書き込みを実行した, **Then** `docs/models/<context-kebab>.md` が作成または更新され、`docs/models/index.md` も同期される
3. **Given** ユーザーが確認せずに拒否した, **When** スキルが処理した, **Then** ファイルへの書き込みは行われず、現状が維持される
4. **Given** 既存ファイルに変更がない, **When** スキルが差分計算した, **Then** 「変更なし」と報告してファイルを上書きしない

---

### User Story 3 - ユビキタス言語との連携ブートストラップ (Priority: P3)

プロジェクトに `docs/ubiquitous-language.md` が存在するとき、スキルはそのエントリをドメインモデル候補として読み込み、ユーザーに提案する。

**Why this priority**: UL スキルと連携することで語彙の重複入力を防ぎ、一貫性が保てる。UL 非存在時も独立動作するため P3 とする。

**Independent Test**: `docs/ubiquitous-language.md` に「注文」エントリがある状態でスキルを起動し、「注文」が集約候補として提案されれば独立検証可能。

**Acceptance Scenarios**:

1. **Given** `docs/ubiquitous-language.md` が存在する, **When** スキルを起動した, **Then** UL の各エントリがドメインモデル候補として提案される
2. **Given** `docs/ubiquitous-language.md` が存在しない, **When** スキルを起動した, **Then** UL なしの独立ブートストラップモードで動作し、エラーにならない
3. **Given** UL に記載済みのエントリを再提案した, **When** ユーザーが確認した, **Then** 既存エントリを重複追加せず、存在済みとして扱う

---

### User Story 4 - インデックスで全コンテキストを一元管理する (Priority: P4)

複数の境界づけられたコンテキストが存在するとき、`docs/models/index.md` が常に全コンテキストファイルへのリンクを持ち、コンテキスト間の関係を記述する。

**Why this priority**: 単一コンテキストで価値が完結するため、マルチコンテキストのインデックス管理は P4 とする。

**Independent Test**: 2 つ以上のコンテキストファイルが存在するとき、`index.md` が両方へのリンクを含んでいれば独立検証可能。

**Acceptance Scenarios**:

1. **Given** 新しいコンテキストファイルが作成された, **When** スキルが書き込みを完了した, **Then** `docs/models/index.md` に新コンテキストへのリンクが追加される
2. **Given** 2 つのコンテキストが存在する, **When** コンテキスト間の関係をユーザーが記述した, **Then** index.md にコンテキスト間関係が反映される
3. **Given** コンテキストファイルが削除された, **When** スキルが次回実行された, **Then** index.md から削除されたコンテキストのリンクが除去される

---

### Edge Cases

- コンテキスト名が競合した場合（同名の異なるコンテキスト）はどう扱うか？
- 既存のコンテキストファイルに別の書式で記述されている場合、スキルはどう対応するか？
- UL に記載のあるドメインイベントと、モデルファイルのドメインイベントが食い違った場合の権威（どちらが正）は？
- 同一ターンで複数コンテキストに跨る語彙が登場した場合の振り分けは？

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: スキルは会話中の DDD 語彙パターン（集約・エンティティ・値オブジェクト・ドメインイベント・不変条件）を受動的に検出し、候補キューに追加しなければならない
- **FR-002**: 候補キューは前のターンで新規候補がなかった時点で提示し、それ以外は会話を中断してはならない
- **FR-003**: スキルは明示コマンドで呼び出せ、指定コンテキストのモデルファイルを生成・更新しなければならない
- **FR-004**: モデルファイルは Mermaid classDiagram・集約テーブル・エンティティテーブル・値オブジェクトテーブル・ドメインイベントテーブル・不変条件リストを含まなければならない
- **FR-005**: Mermaid 図はテーブルから機械生成しなければならない。テーブルと図が乖離した場合は再生成を提案しなければならない
- **FR-006**: ファイル書き込み前に必ず差分を提示し、明示的な確認なしに書き込んではならない
- **FR-007**: 1 つの境界づけられたコンテキスト = 1 ファイル (`docs/models/<context-kebab>.md`) を厳守しなければならない
- **FR-008**: `docs/models/index.md` は全コンテキストファイルを常に反映し、ファイル作成・更新・削除のたびに同期されなければならない
- **FR-009**: `docs/ubiquitous-language.md` が存在する場合、スキルはそれを読み込みドメインモデル候補として提案しなければならない
- **FR-010**: コンテキスト間で概念が衝突した場合、分離か改名かをユーザーに提示しなければならない。サイレントマージは禁止
- **FR-011**: すべての出力（質問・差分・ファイル内容）は日本語で提示しなければならない
- **FR-012**: 集約テーブルは 7 フィールド（集約名・集約ルート・責務・境界・不変条件・ライフサイクル・実装名）を持たなければならない。ただし、コードとの対応がまだ存在しない段階では「実装名」フィールドを空欄のままにできる

### Key Entities

- **Bounded Context（境界づけられたコンテキスト）**: ドメインモデルの適用範囲; 1 ファイルに対応
- **Aggregate（集約）**: 整合性境界を持つエンティティ群のルート; 不変条件を管理
- **Entity（エンティティ）**: 識別子で同一性を判定するドメインオブジェクト; ライフサイクルを持つ
- **Value Object（値オブジェクト）**: 値の等価性で同一性を判定する不変オブジェクト
- **Domain Event（ドメインイベント）**: ビジネス上の重要な出来事の記録; 過去形で命名
- **Invariant（不変条件）**: 集約が常に維持しなければならないビジネスルール

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 会話中に登場した DDD 語彙の 100% がキューに追加される（検出漏れゼロ）
- **SC-002**: ユーザーの明示確認なしにモデルファイルが書き込まれることがない（確認前書き込み件数 = 0）
- **SC-003**: Mermaid 図とテーブルの内容が常に一致しており、乖離を自動検出して提案する
- **SC-004**: コンテキストファイルの作成・更新・削除後、`index.md` が即座に同期される（手動操作不要）
- **SC-005**: `docs/ubiquitous-language.md` が存在する場合、UL の全エントリがモデル候補として提示される
- **SC-006**: コンテキスト間の概念衝突が発生した場合、100% の確率でユーザーに解決を求め、サイレントマージが発生しない

## Assumptions

- ユーザーは DDD の基本概念（集約・エンティティ・値オブジェクト・ドメインイベント）に一定の知識を持つ開発者またはアーキテクトである
- スキルは Claude Code の会話環境内で動作し、ファイルシステムへの読み書き権限がある
- プロジェクトによっては `docs/ubiquitous-language.md` が存在しない場合もあり、その場合は独立動作する
- 単一プロジェクト内に複数の境界づけられたコンテキストが存在しうる
- 初回ブートストラップ（既存モデルなし）と既存モデルへの増分更新の両モードをサポートする
- ドメインイベントの命名と定義は `docs/ubiquitous-language.md` が唯一の正（Single Source of Truth）。ドメインモデルスキルはドメインイベントを UL から読み込み、モデルファイル内ではペイロード・後続処理などの構造情報のみ追記する。UL への書き込みは行わない
