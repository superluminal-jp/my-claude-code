# Phase 1 Data Model: Codex CLI Guardrail Implementation

This feature's "data" is a set of files, scripts, and configuration entries, not application state. The entities below describe the structure each artifact must follow so it satisfies `spec.md`'s Functional Requirements and Success Criteria.

## AGENTS.md Entry

One of the 12 prose-appropriate decision-record items carried in `AGENTS.md`.

| Field | Type | Notes |
|---|---|---|
| `item_id` | string (Q1, Q2, Q4, Q5, Q7–Q14) | FK to `specs/012-cross-agent-guardrail-migration/decision-record.md`'s Migration Item |
| `content` | markdown | The guidance text itself; genericized per FR-003 for Q2, mechanism-agnostic per FR-005 for Q13/Q14 |
| `enforcement_note` | string \| null | Populated only for Q7, Q9, Q10 — states "Codex CLI enforces this via hook" (FR-002); null for the 9 purely-prose items |
| `canonical` | bool | `true` only for Q4 — its content is authored here and imported elsewhere (FR-023), not the reverse |

12 instances: Q1, Q2, Q4, Q5, Q7, Q8, Q9, Q10, Q11, Q12, Q13, Q14. Q3 and Q6 are not `AGENTS.md` entries (see Skill Symlink and Shared Guardrail Script below).

## Shared Guardrail Script

Single source of truth for one existing hook's matching/formatting logic, extracted so both Claude Code and Codex CLI can call it.

| Field | Type | Notes |
|---|---|---|
| `script_id` | enum (`destructive-command`, `pre-edit-block`, `post-edit-format`) | One per FR-006/FR-016/FR-019 |
| `source_hook` | path | `.claude/hooks/pre-bash.sh` \| `pre-edit.sh` \| `post-edit-format.sh` — the hook this script's logic is extracted from |
| `path` | path | `scripts/guardrails/<script_id>.sh` |
| `input_contract` | see `contracts/guardrail-script-io.md` | Tool-agnostic stdin shape |
| `output_contract` | see `contracts/guardrail-script-io.md` | `destructive-command`/`pre-edit-block`: `{decision: allow\|deny\|ask, reason}`. `post-edit-format`: no decision (automation only, never blocks) |
| `consumers` | list | Always exactly 2: the refactored Claude Code hook (`source_hook`, now a thin wrapper) and one Codex CLI Adapter |

3 instances.

## Codex CLI Adapter

Thin translation layer between one Shared Guardrail Script's contract and Codex CLI's hook contract (R1).

| Field | Type | Notes |
|---|---|---|
| `adapter_id` | enum (`destructive-command-adapter`, `pre-edit-adapter`, `post-edit-adapter`) | |
| `path` | path | `.codex/hooks/<adapter_id>.sh`, deployed to `~/.codex/hooks/<adapter_id>.sh` by `install.sh` (R6) |
| `wraps` | FK → Shared Guardrail Script | 1:1 |
| `hook_event` | enum (`PreToolUse`, `PostToolUse`) | `PreToolUse` for `destructive-command-adapter`/`pre-edit-adapter`; `PostToolUse` for `post-edit-adapter` |
| `matcher` | string | `"Bash"` (destructive-command); `"apply_patch\|Edit\|Write"` (pre-edit, post-edit) |
| `registration` | path | `~/.codex/config.toml`'s `[hooks]` table (R1), written by `install.sh` |

3 instances. No Cursor adapter exists in this feature (spec FR-015).

## Skill Symlink

One of the 9 custom skills, made discoverable to Codex CLI at two tiers (R2, R5).

| Field | Type | Notes |
|---|---|---|
| `skill_name` | string (adr, advisor, clarifier, coder, domain-model, minto-builder, minto-reviewer, minto-rewriter, ubiquitous-language) | |
| `source` | path | `.claude/skills/<skill_name>/SKILL.md` — the single authored copy |
| `repo_local_link` | path | `.agents/skills/<skill_name>` → `../../.claude/skills/<skill_name>` (relative symlink, committed to git) |
| `global_link` | path | `~/.agents/skills/<skill_name>` → `~/.claude/skills/<skill_name>` (created by `install.sh`, after its existing skills sync step) |

9 instances. The 15 `speckit-*` entries already present under `.agents/skills/` are a separate, pre-existing, gitignored mechanism (per-project, regenerated by `speckit-expand-update.sh`) and are not modeled here — this feature does not touch them.

## `install.sh` Deployment Step

One of the new global-deployment actions this feature adds to the existing installer (FR-025–028).

| Field | Type | Notes |
|---|---|---|
| `step_id` | enum (`agents-md`, `agents-skills`, `codex-hooks`) | |
| `source` | path | `AGENTS.md`; `.claude/skills/<name>` (via the pre-existing skills sync, R5); `.codex/hooks/*.sh` |
| `target` | path | `~/.codex/AGENTS.md`; `~/.agents/skills/<name>`; `~/.codex/hooks/*.sh` + `~/.codex/config.toml` `[hooks]` entries |
| `mechanism` | enum (`copy`, `symlink`, `config-merge`) | `agents-md`: copy (`sync_path`-style, matches R4). `agents-skills`: symlink (R5). `codex-hooks`: copy the adapter scripts (`sync_path`-style) + merge/upsert their entries into `~/.codex/config.toml`'s `[hooks]` table (idempotent — re-running must not duplicate entries) |
| `idempotent` | bool | `true` for all three, required by spec SC-011 |

3 instances, each a new step appended to `install.sh`'s existing numbered sequence (after step 1, "Sync managed .claude paths").

## `.claude/rules/mcp.md` Import

The single Claude Code-side file this feature restructures rather than creates (FR-023/024/028).

| Field | Type | Notes |
|---|---|---|
| `import_target` | path | The installed global `AGENTS.md` (`~/.codex/AGENTS.md`), per R7 — not a repository-relative path |
| `import_syntax` | string | `@path` syntax per Claude Code's documented import mechanism; exact resolution against a `$HOME`-relative target is R7's one unresolved item, verified at implementation time (FR-024) |
| `fallback` | path \| null | If FR-024's verification fails: `.claude/rules/mcp.md` keeps its pre-existing standalone MCP catalog content (the file as it exists before this feature), and this field records that the import was not adopted |

1 instance.
