# Phase 0 Research: Codex CLI Guardrail Implementation

**Status**: No `NEEDS CLARIFICATION` markers remain in `spec.md` — every ambiguity was resolved during this session's `/speckit-specify` and `/speckit-clarify` passes (see `spec.md`'s `## Clarifications`). This document consolidates the technical decisions those resolutions depend on, so implementation has one place to check before writing code, and records which claims are still Medium-confidence and must be re-verified live (per spec FR-010/FR-022/FR-024/FR-028).

## R1 — Codex CLI hook registration format and location

- **Decision**: Codex CLI hooks are registered under a `[hooks]` table in `config.toml` (or an equivalent `hooks.json`), keyed by event name (`PreToolUse`, `PostToolUse`, …), each entry carrying a `matcher` (e.g. `"Bash"`, `"apply_patch"`, `"Edit"`, `"Write"`) and a `hooks` array of `{type = "command", command = "...", timeout = ...}` entries. Global scope is `~/.codex/config.toml`; project scope is `<repo>/.codex/config.toml`, loaded only when the project directory is trusted. This feature registers at **global scope**, per spec's Clarifications (mirrors `.claude/settings.json`'s existing global hook wiring).
- **Rationale**: Directly documented at [learn.chatgpt.com/docs/hooks](https://learn.chatgpt.com/docs/hooks), consulted this session. The JSON/TOML shape (`matcher` + `hooks: [{type: "command", command, timeout}]`) is close enough to Claude Code's own `settings.json` hook shape that the adapter scripts need no unusual translation layer beyond the response-JSON shape (see R7 and `contracts/guardrail-script-io.md`).
- **Alternatives considered**: Project-scoped registration (`<repo>/.codex/config.toml`) — rejected per the spec's Clarifications decision to match `.claude/`'s existing global-install model, not scope this feature to one repository.
- **Confidence**: Medium — this session's source is a live documentation fetch, not a primary-source-verified Codex CLI session. FR-010 (destructive-command adapter) and FR-022 (pre-edit/post-edit adapters) already require re-verifying the exact event coverage and response shape against a live Codex CLI session before those adapters are considered complete; this applies equally to the registration format itself.

## R2 — Codex CLI skill discovery

- **Decision**: Codex CLI scans `.agents/skills/<name>/SKILL.md` from the current working directory up to the repository root (project scope), and additionally `$HOME/.agents/skills/<name>/SKILL.md` (user/global scope). This feature populates **both**: repo-local `.agents/skills/<name>` as symlinks to `.claude/skills/<name>` (so this repository's own working tree is self-consistent and useful even before installation), and `~/.agents/skills/<name>` via `install.sh`, for every other project on the machine.
- **Rationale**: [learn.chatgpt.com/docs/build-skills](https://learn.chatgpt.com/docs/build-skills), corroborated by [customization/overview](https://learn.chatgpt.com/docs/customization/overview). This repository already relies on the project-scope tier for its 15 `speckit-*` skills (populated by `speckit-expand-update.sh`, gitignored per `docs/adr/0001-remove-vendored-speckit-skills.md`) — this feature adds the user-scope tier for the 9 hand-authored skills, which is the tier that actually achieves "every project on the machine," matching the spec's global-deployment Clarification.
- **Alternatives considered**: Repo-local symlinks only (rejected — doesn't reach other projects, contradicting the global-deployment decision); a generated copy instead of a symlink at either tier (rejected per spec FR-014 — symlinks eliminate drift by construction, copies only detect it).
- **Confidence**: High for the discovery mechanism itself (corroborated by two independent doc pages); Medium for the exact global-tier path (`$HOME/.agents/skills`) until spot-checked against a live Codex CLI session, per spec's Assumptions.

## R3 — AGENTS.md discovery, precedence, and size budget

- **Decision**: `AGENTS.md` is discovered at global scope (`~/.codex/AGENTS.md`, checked before `~/.codex/AGENTS.override.md`) and project scope (repo root down to cwd, `AGENTS.override.md` taking precedence per directory), concatenated root-to-cwd with closer files overriding. Combined content is capped at 32 KiB by default (`project_doc_max_bytes`); content beyond the cap is silently dropped. This feature authors one file at the repository root and deploys it to the **global** tier (`~/.codex/AGENTS.md`) via `install.sh`, per the spec's global-deployment Clarification — no project-scope `AGENTS.md` or `AGENTS.override.md` is introduced.
- **Rationale**: [learn.chatgpt.com/docs/agent-configuration/agents-md](https://learn.chatgpt.com/docs/agent-configuration/agents-md), consulted this session. A single global file is sufficient because this repository's guardrail content (the 12 `AGENTS.md` entries) is universal advice, not directory-specific.
- **Alternatives considered**: Project-scope-only deployment (rejected — contradicts the global-deployment Clarification); splitting content across multiple nested `AGENTS.md` files (rejected — no directory-specific guidance exists to justify the split, and it would complicate the 32 KiB budget tracking for no benefit).
- **Confidence**: High (this is the single most corroborated finding across this session's research — confirmed by both vendor-quoted claims and this repository's own working precedent for `.agents/skills`).

## R4 — `install.sh`'s existing sync pattern

- **Decision**: `install.sh` syncs managed paths via a `sync_path()` helper: `rm -rf "$dst"; cp -R "$src" "$dst"` (directories) or `cp "$src" "$dst"` (files) — a blunt, idempotent full-overwrite, not a diff/merge. It special-cases stripping `speckit-*` entries from the synced `.claude/skills/` (since those are per-project generated artifacts, never centrally managed). This feature's new deployment steps (`AGENTS.md`, `.agents/skills/` symlinks, Codex CLI hook registration) follow the **same** `sync_path()`-style pattern rather than introducing a different install mechanism, per the spec's Assumptions.
- **Rationale**: Read directly from `install.sh` in this session. Matching the existing pattern keeps the installer's behavior predictable (one mental model: "re-running always makes the target match the source exactly") and reuses a mechanism already proven idempotent.
- **Alternatives considered**: A separate, more sophisticated sync tool for the new Codex CLI artifacts (rejected — unjustified complexity for four straightforward deployment steps; violates this repo's own simplicity conventions).
- **Confidence**: High — this is this repository's own source code, read directly, not a secondary claim.

## R5 — Symlink strategy for the 9 custom skills

- **Decision**: Two symlink hops, not a copy chain. (1) Repo-local: `.agents/skills/<name>` → `../../.claude/skills/<name>` (relative symlink, committed to git — git supports symlinks natively). (2) Global, via `install.sh`: after `install.sh`'s existing `sync_path("skills")` step copies `.claude/skills/` to `~/.claude/skills/` (unchanged), a new step symlinks `~/.agents/skills/<name>` → `~/.claude/skills/<name>` for each of the 9 custom names — pointing at the already-synced global copy, not back at the repository's working tree (which may not exist on a machine that only ran the installer from a temporary clone).
- **Rationale**: Satisfies FR-014's "eliminate drift by construction" for both the repo-local working-tree case (User Story 2's independent test) and the global-install case (FR-026), without ever maintaining three independent copies of the same `SKILL.md`.
- **Alternatives considered**: Global symlink pointing directly at the original repository path (rejected — breaks if the repository is deleted or moved after install, unlike Claude Code's own `.claude/skills/`, which is copied, not symlinked, specifically to survive that case); a copy at every tier (rejected per FR-014).
- **Confidence**: High — symlink semantics are well-understood; the only implementation risk is git's own symlink support on the target platform, already implicitly relied upon by nothing else in this repo (new risk surface, worth a quickstart check).

## R6 — Codex CLI adapter script location

- **Decision**: Repo-local adapter scripts live under `.codex/hooks/` (mirroring `.claude/hooks/`'s naming), deployed by `install.sh` to `~/.codex/hooks/` via a new `sync_path`-style step, with `~/.codex/config.toml`'s `[hooks]` table (R1) referencing the installed path (`"$HOME/.codex/hooks/destructive-command-adapter.sh"`), exactly mirroring how `.claude/settings.json`'s hook commands already reference `"$HOME"/.claude/hooks/...`.
- **Rationale**: Directly parallels this repository's existing `.claude/hooks/` → `~/.claude/hooks/` + `settings.json` pattern (R4), so a maintainer already familiar with this repo's Claude Code side finds the Codex CLI side structurally identical.
- **Alternatives considered**: Registering the shared scripts (`scripts/guardrails/*.sh`) directly as Codex CLI hooks, with no separate adapter layer (rejected — the shared scripts' stdin/stdout contract, `contracts/guardrail-script-io.md`, is tool-agnostic by design per FR-006/016/019; a thin adapter is still needed to translate Codex CLI's specific hook request/response JSON shape to and from that contract, matching the same separation Claude Code's own hooks already have from the shared scripts).
- **Confidence**: High for the structural decision (an internally consistent choice this repo controls); Medium for `~/.codex/config.toml`'s exact `[hooks]` field names (inherits R1's confidence).

## R7 — `@path` import target for `.claude/rules/mcp.md` (FR-023/024/028)

- **Decision**: `.claude/rules/mcp.md` imports the installed global `AGENTS.md` (`~/.codex/AGENTS.md`, matching where `.claude/rules/mcp.md` itself resolves to after installation, `~/.claude/rules/mcp.md`) — not a repository-relative path, since the file only has a stable location once both are installed under `$HOME`. The literal `@path` target syntax (e.g. whether `@~/.codex/AGENTS.md` or an absolute `$HOME`-expanded form is required) is the one claim in this feature **not yet confirmed by any source consulted this session** — [features-overview](https://code.claude.com/docs/en/features-overview) confirms `@path` imports exist for `CLAUDE.md`/rules files, but not their exact path-resolution rules for a target outside `.claude/` entirely.
- **Rationale**: The global-deployment Clarification (spec) requires this file to work correctly *after* installation, not only when read from this repository's own working tree — so the import target must be the installed path, not a repo-relative one.
- **Alternatives considered**: A repo-relative import (`@../../AGENTS.md`) that works only inside this repository's own working tree (rejected — breaks for every other installed project, and even inside this repo, Claude Code sessions here read the installed `~/.claude/` copy per R4/R6's precedent, not the working tree, once `install.sh` has run).
- **Confidence**: Low-Medium — this is the one open technical risk in this feature. FR-024 already requires live verification before FR-023 is considered complete; if `@path` cannot resolve a `$HOME`-relative or absolute target, the documented fallback (spec FR-024) applies: `.claude/rules/mcp.md` keeps its current standalone content, and User Story 6 / SC-010 is marked as not achieved, with the gap documented rather than silently worked around.

## Summary

No further research is required before Phase 1. R1–R6 support implementation with High-to-Medium confidence; R7 is the one item where implementation must be prepared for its designed fallback to trigger. Where confidence is Medium or lower, the corresponding spec-level FR (FR-010, FR-022, FR-024, FR-028) already mandates live re-verification before the affected work is considered complete — this document does not relax those requirements, it only records what each one is checking.
