**Status**: Draft

**Follow-up correction 2 (same session)**: The maintainer identified, via [learn.chatgpt.com/docs/hooks](https://learn.chatgpt.com/docs/hooks), that Codex CLI's `PreToolUse`/`PostToolUse` hooks cover `apply_patch`/`Edit`/`Write` (file-edit tools), not just Bash — contradicting `specs/012-cross-agent-guardrail-migration/research.md`'s R4 ("no tool has a pre-edit... file-write block") for Codex CLI specifically, and matching `.claude/hooks/pre-edit.sh`'s and `post-edit-format.sh`'s own stdin/stdout contract closely enough that reuse is straightforward. On confirmation, Q9 (main/master block) and Q10 (`.git/` block) are upgraded from `unify_weak` (prose-only) to real enforcement for Codex CLI, and Q7 (post-edit auto-format) is upgraded the same way for Codex CLI.

**Scope narrowing (same session)**: The maintainer decided Cursor support is explicitly **out of scope** for this spec. All Cursor-targeting work originally planned here (a `beforeShellExecution` destructive-command adapter, and any Cursor-specific claims about skill discovery or pre-edit/post-edit hooks) is removed from this feature. `AGENTS.md` remains a generic, tool-agnostic file — if Cursor happens to also read it, that is incidental, not something this feature builds for or verifies. A future spec may extend Cursor coverage using the same shared-script pattern this feature establishes (each shared script's inputs/outputs are already tool-agnostic).

**Follow-up references (same session)**: The maintainer supplied five further official Codex customization docs — [customization/overview](https://learn.chatgpt.com/docs/customization/overview), [customization/memories](https://learn.chatgpt.com/docs/customization/memories?surface=app), [agent-configuration/agents-md](https://learn.chatgpt.com/docs/agent-configuration/agents-md), [agent-configuration/subagents](https://learn.chatgpt.com/docs/agent-configuration/subagents?surface=app), and [agent-configuration/rules](https://learn.chatgpt.com/docs/agent-configuration/rules) — for citation and incorporation. Findings, each cited at point of use below:
- **AGENTS.md** has a **32 KiB default size budget** (`project_doc_max_bytes`) and a global→project, root→cwd discovery/merge hierarchy with per-directory override files (`AGENTS.override.md`) — refines FR-001/FR-011 and R5 with concrete constraints spec 012's research did not have.
- **Memories** (`~/.codex/memories/`) is user-scoped and auto-generated from session history, not repository-controllable — this **confirms** (does not change) Q13's original mechanism-agnostic prose treatment, since there is no repo-side integration point to build against.
- **Skills** documentation corroborates FR-013's `.agents/skills` design (`SKILL.md` required, optional `scripts/`/`references/`/`assets/`) with no new constraint affecting this feature's scope.
- **Rules** (Starlark `.rules` files, `prefix_rule()`, decisions `allow`/`prompt`/`forbidden`, scanned from `~/.codex/rules/` and `<repo>/.codex/rules/`) is a distinct, more native command-permission mechanism than the `PreToolUse` shell-script adapter this spec already plans for Q6 (FR-006/007) — but its prefix/union matching appears less expressive than `pre-bash.sh`'s embedded regex matching (e.g. detecting `rm -rf` after `&&` mid-command), so it is not a drop-in replacement. Flagged as a planning-time evaluation, not a spec change (see Assumptions) — this spec does not mandate Rules-based implementation.
- **Subagents** (TOML files under `~/.codex/agents/` or `.codex/agents/` with `name`/`description`/`developer_instructions`) is a real, concrete mechanism beyond the mechanism-agnostic prose spec 012's Q14 verdict already satisfies. Defining actual Codex subagent configs (e.g. an `explorer`-equivalent) would be a genuine enhancement but exceeds what Q14's recorded verdict requires — flagged as an optional future extension, not added to this spec's scope (see Assumptions).

**Purpose clarification and Claude Code-side references (same session)**: The maintainer stated this spec's purpose directly: unify shareable resources so the *same* configuration applies to both Codex CLI and Claude Code, not just parallel configurations with shared underlying logic. The maintainer supplied Claude Code's own official docs — [features-overview](https://code.claude.com/docs/en/features-overview) and [claude-directory](https://code.claude.com/docs/en/claude-directory) — confirming the Claude Code-side facts this goal depends on:
- Claude Code does **not** read `AGENTS.md` at all — it reads `CLAUDE.md` (root or `.claude/CLAUDE.md`) and `.claude/rules/*.md` exclusively. A single literal file readable natively by both tools does not exist for prose guidance; `AGENTS.md` and `CLAUDE.md`/`.claude/rules/` remain two files by necessity.
- `.claude/skills/<name>/SKILL.md` and Codex's `.agents/skills/<name>/SKILL.md` use the **same directory-per-skill, `SKILL.md`-required structure** — the strongest case in this feature for literal, zero-duplication sharing (see FR-014's revision below, symlinks instead of copies).
- `CLAUDE.md` and skills support `@path` imports ("Can include files: Yes, with `@path` imports," per features-overview); `.claude/rules/*.md` files load the same way `CLAUDE.md` does. This raises the possibility of `.claude/rules/*.md` importing shared prose from `AGENTS.md` via `@path` for the items that are already tool-agnostic (Q1, Q4, and others with no Claude Code-specific wording) — turning `AGENTS.md` into the canonical source for those items instead of a separately hand-maintained transcription. **Adopted for Q4** — see User Story 6, FR-023/024, SC-010.
- Claude Code's `.claude/rules/` and Codex's "Rules" (Starlark, [agent-configuration/rules](https://learn.chatgpt.com/docs/agent-configuration/rules)) share a name but are **opposite kinds of mechanism**: Claude Code's rules are prose guidance ("like CLAUDE.md, rules are guidance Claude reads, not configuration Claude Code enforces," per claude-directory) — the closer Claude Code analog to Codex's enforced Rules is actually `.claude/settings.json`'s `permissions.allow`/`permissions.deny` glob patterns (e.g. `"Bash(rm -rf *)"`), not `.claude/rules/*.md`. This terminology clash is worth stating explicitly so future readers of this spec don't conflate the two.

**MCP connection-config references (same session)**: The maintainer supplied [mcp-quickstart](https://code.claude.com/docs/en/mcp-quickstart#connect-to-mcp-servers) and [customization/overview#mcp](https://learn.chatgpt.com/docs/customization/overview#mcp) specifically for Q4's MCP content. Findings:
- Claude Code's MCP server *connections* live in project-scoped `.mcp.json` (JSON: `{"mcpServers": {"<name>": {"type": "http"|"stdio", "url" | "command"+"args", ...}}}`), already referenced by this repository's own `CLAUDE.md` ("Project MCP definitions live in `.mcp.json`").
- Codex CLI's MCP server *connections* live in TOML `config.toml` (`~/.codex/config.toml` or project `.codex/config.toml`), with a different field set (`command`/`args`/`env`/`env_vars` for stdio; `url`/`auth`/`bearer_token_env_var` for HTTP) — confirmed via [Codex's MCP extend docs](https://learn.chatgpt.com/codex/extend/mcp). Codex CLI does **not** read `.mcp.json`; the two connection-config formats are incompatible by construction (JSON vs. TOML, different field names), so unifying the *connection* configs themselves is not possible and is not attempted by this feature.
- This does not affect User Story 6 / Q4: `.claude/rules/mcp.md`'s content is prose *usage guidance* ("which server to use for which kind of question"), not the connection config — that prose is tool-agnostic and unaffected by the connection-format difference, so the `@path`-import dedup plan (FR-023/024) stands unchanged.

Spec 012's own decision record is not rewritten by any of the above (already committed); this spec absorbs all of it since it is still in Draft.

## Overview

Spec 012 answered, for each of 14 Claude Code-specific `.claude/` hooks and rules, whether it should be unified for OpenAI Codex CLI and Cursor compatibility — and if so, at what strength (full enforcement vs. prose-only note). That decision record is final and committed. **This feature is the follow-up build that turns the Codex CLI-relevant decisions into files that actually exist**: today there is no `AGENTS.md` in the repository, and none of `.claude/hooks/pre-bash.sh`, `pre-edit.sh`, or `post-edit-format.sh` has a Codex CLI counterpart.

This repository is itself a distributable, user-level Claude Code configuration: its own `README.md` states `.claude/` "is designed to be copied wholesale to `~/.claude/`" via `install.sh`, applying across every project on the machine, and `.claude/settings.json`'s hook commands already reference the installed global path (`"$HOME"/.claude/hooks/...`), not a repository-relative one. Per the Clarifications session above, this feature follows the same model: `AGENTS.md`, `.agents/skills/`, and the shared guardrail scripts are authored in this repository but **deployed globally** by an extended `install.sh`, so Codex CLI sessions in any project on the machine get the same guardrails and guidance Claude Code sessions already get everywhere.

This specification defines what must exist for a maintainer working in Codex CLI — after installing this repository's configuration — to receive (a) the same baseline guidance Claude Code sessions already get, (b) the same destructive-command protection Claude Code sessions already get, (c) native discovery of this repository's own skills, and (d) the same pre-edit blocking and post-edit formatting Claude Code sessions already get. It does not redefine which items are unified or at what strength for Claude Code itself — those are fixed inputs from `specs/012-cross-agent-guardrail-migration/decision-record.md`, except where explicitly superseded above (Q3, Q7, Q9, Q10 — each upgraded for Codex CLI only). Cursor is out of scope entirely (see Scope narrowing above).

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Shared baseline guidance via AGENTS.md (Priority: P1)

A maintainer who has installed this repository's configuration (`install.sh`) opens any project in Codex CLI instead of Claude Code. They want the same baseline guidance Claude Code sessions get from `.claude/rules/*` — the prose-appropriate items spec 012 decided to unify — available to them everywhere, without opening `.claude/` at all.

**Why this priority**: This is the highest-value, lowest-risk deliverable — it is markdown content only, requires no new hook mechanism, and depends on nothing else in this feature. It alone gives Codex CLI sessions parity with Claude Code's non-enforcement guidance.

**Independent Test**: With only `AGENTS.md` created (no adapters yet), a reviewer opens the file and confirms all 12 prose-appropriate items from the decision record are present, each phrased at the strength (full, or Codex-enforced, or weak) that applies, independent of whether any other user story in this feature has shipped.

**Acceptance Scenarios**:

1. **Given** decision-record.md's `unify_full` verdicts for Q1, Q2, Q4, and Q14, **When** `AGENTS.md` is created, **Then** it contains an entry for each, with Q2's tool references genericized (no `Read`/`Edit`/`Grep`-style Claude Code tool names). Q3 (skill routing) is **not** carried as `AGENTS.md` prose in this feature — see User Story 2, which supersedes the original prose-transcription treatment with native sync for Codex CLI.
2. **Given** decision-record.md's `unify_weak` verdicts for Q5, Q8, Q11, Q12, and Q13, **When** `AGENTS.md` is created, **Then** it contains a prose-only entry for each, worded as a note or request rather than a guarantee, and Q13's entry names no Claude Code-specific mechanism (e.g. does not say "Claude Memory").
3. **Given** Q7, Q9, and Q10, upgraded to real enforcement for Codex CLI by User Stories 4–5, **When** `AGENTS.md` is created, **Then** its entry for each states plainly that Codex CLI enforces the item via hook — it does not read as a mere unenforced note.
4. **Given** `AGENTS.md` does not exist yet in this repository, **When** it is created, **Then** it is a new file at the repository root (not nested under `.claude/` or `.codex/`), matching the shared-file premise `specs/012-cross-agent-guardrail-migration/research.md`'s R5 finding relies on.
5. **Given** a hypothetical future run where `AGENTS.md` already exists with unrelated maintainer content, **When** this feature's rollout runs, **Then** the existing content is surfaced to the maintainer before any overwrite, not silently replaced.

### User Story 2 - Native skill discovery for Codex CLI via .agents/skills (Priority: P1)

A maintainer working in Codex CLI, in any project, wants this repository's own skills (`adr`, `advisor`, `clarifier`, `coder`, `domain-model`, `minto-builder`, `minto-reviewer`, `minto-rewriter`, `ubiquitous-language`) to trigger the same way they do in Claude Code everywhere — not as a lossy prose summary, but as the real `SKILL.md` files, discovered the same global way Codex CLI already discovers user-level skills (per `$HOME/.agents/skills`).

**Why this priority**: Codex CLI natively scans `.agents/skills/<name>/SKILL.md` from the working directory up to the repository root (confirmed via [learn.chatgpt.com/docs/build-skills](https://learn.chatgpt.com/docs/build-skills)); this repository already relies on that exact mechanism for its 15 `speckit-*` skills via the existing `speckit-expand-update.sh` sync. Extending the same, already-proven mechanism to the 9 remaining skills is zero-drift and lower-risk than the prose-routing-table approach Q3 originally specified, so it supersedes that approach and ranks at P1 alongside User Story 1.

**Independent Test**: With this story implemented alone (no `AGENTS.md`, no adapters), a reviewer runs a request in a Codex CLI session that should trigger one of the 9 custom skills (e.g. a decision/trade-off request that should trigger `advisor`) and confirms Codex CLI reads that skill's real `SKILL.md` from `.agents/skills/advisor/SKILL.md`, independent of every other user story in this feature.

**Acceptance Scenarios**:

1. **Given** the 9 custom-authored skills currently exist only under `.claude/skills/`, **When** this story is implemented, **Then** each of the 9 also exists under `.agents/skills/<name>/SKILL.md`, matching the directory-per-skill structure Codex CLI's documentation specifies (a directory containing `SKILL.md` plus any optional scripts/references).
2. **Given** a custom skill's `.claude/skills/<name>/SKILL.md` is edited after this story ships, **When** the next sync point is reached, **Then** `.agents/skills/<name>/SKILL.md` reflects the same content — the two must not be allowed to silently drift apart.

### User Story 3 - Real destructive-command blocking for Codex CLI (Priority: P2)

A maintainer runs a destructive shell command (force push, `rm -rf ~`, `curl | bash`, etc.) from a Codex CLI session in this repository. Today nothing stops them — only Claude Code has this protection. They want the same block/ask outcome Claude Code already gives.

**Why this priority**: Decision-record.md flags Q6 as the item with the highest security value of all 14 — real enforcement, not documentation — but it requires new code (a shared matcher script plus a tool adapter), so it ranks below the zero-risk documentation wins in User Stories 1–2.

**Independent Test**: With `.claude/hooks/pre-bash.sh`'s matching logic extracted into a shared script and a Codex CLI adapter wired to it, run each destructive-command category (force push, `git reset --hard`, `git clean -f`, `rm -rf` against root/home/cwd, other `rm -rf`, `mkfs`/`dd`/fork-bomb, `curl|bash`, non-HTTPS request, credential-path read, credential-path write, global package install, `sudo`) through a Codex CLI session and confirm the same allow/deny/ask outcome Claude Code produces for that same command.

**Acceptance Scenarios**:

1. **Given** a command matching a currently-hard-blocked category (e.g. `git push --force`, `rm -rf /`), **When** it is run through the Codex CLI adapter, **Then** the command is denied before execution.
2. **Given** a command matching a currently-"ask"-routed category (`rm -rf` outside root/home/cwd, `sudo`), **When** it is run through the Codex CLI adapter, **Then** the adapter either routes it to confirmation (if Codex CLI's hook contract supports a three-way decision) or denies it by default (if it does not) — never silently allows it.
3. **Given** a benign command matching none of the categories, **When** it is run through the Codex CLI adapter, **Then** it is allowed.
4. **Given** `research.md`'s R1 (Codex CLI `PreToolUse` scope) is flagged Medium-confidence, **When** the adapter is implemented, **Then** R1's claim is re-verified against Codex CLI's primary documentation before the adapter is considered complete.

### User Story 4 - Real pre-edit blocking for Codex CLI (Priority: P2)

A maintainer working in Codex CLI attempts to edit a file directly under `.git/`, or attempts any edit while the current branch is `main`/`master`. Today nothing stops them — only Claude Code has this protection (`.claude/hooks/pre-edit.sh`). They want the same block Claude Code already gives, since Codex CLI's `PreToolUse` hook has now been confirmed to intercept `apply_patch`/`Edit`/`Write` calls before they complete — the exact "before the fact" coverage `specs/012-cross-agent-guardrail-migration/research.md`'s R4 assumed did not exist for any tool.

**Why this priority**: This upgrades two items (Q9, Q10) from documentation-only notes to real enforcement at comparatively low cost, since Codex CLI's `PreToolUse` input/output contract (JSON stdin with `tool_name`/`tool_input`, exit-code/JSON-based block) closely mirrors `.claude/hooks/pre-edit.sh`'s own contract. Ranked P2 — same tier as User Story 3 — because, unlike User Stories 1–2's zero-risk documentation wins, this requires new adapter code and the re-verification FR-022 requires.

**Independent Test**: With `.claude/hooks/pre-edit.sh`'s `.git/`-path and main/master-branch matching logic extracted into a shared script and a Codex CLI `PreToolUse` adapter (scoped to `apply_patch`/`Edit`/`Write`) wired to it, attempt an edit under `.git/` and an edit while on `main` through a Codex CLI session, and confirm both are denied before the edit completes.

**Acceptance Scenarios**:

1. **Given** a file path under `.git/`, **When** an edit is attempted through the Codex CLI adapter, **Then** the edit is denied before it completes, matching `.claude/hooks/pre-edit.sh`'s current behavior.
2. **Given** the current branch is `main` or `master`, **When** any edit is attempted through the Codex CLI adapter, **Then** the edit is denied before it completes, matching `.claude/hooks/pre-edit.sh`'s current behavior.
3. **Given** a file path not under `.git/` and a non-`main`/`master` branch, **When** an edit is attempted, **Then** it is allowed (no false-positive blocking).
4. **Given** [learn.chatgpt.com/docs/hooks](https://learn.chatgpt.com/docs/hooks)'s claim that `PreToolUse` covers `apply_patch`/`Edit`/`Write` is a secondary confirmation gathered mid-session, **When** the adapter is implemented, **Then** the claim is re-verified against a live Codex CLI session before the adapter is considered complete (FR-022).

### User Story 5 - Real post-edit auto-format for Codex CLI (Priority: P3)

A maintainer edits a `.sh`, `.yaml`/`.yml`, or `.json` file through Codex CLI. Today nothing runs `shfmt`/`shellcheck`/`yamllint`/`jq` afterward — only Claude Code sessions get that via `.claude/hooks/post-edit-format.sh`. They want the same automatic formatting/linting, since Codex CLI's `PostToolUse` hook has now been confirmed to cover the same edit tools as `PreToolUse`.

**Why this priority**: Lower priority than User Story 4 because it is automation quality-of-life (silent formatting), not a security guardrail — no destructive action is prevented if this story is deferred. It depends on nothing this feature hasn't already built (the shared-script extraction pattern from User Stories 3–4).

**Independent Test**: With `.claude/hooks/post-edit-format.sh`'s per-extension formatting logic extracted into a shared script and a Codex CLI `PostToolUse` adapter wired to it, edit a `.sh` file through a Codex CLI session and confirm `shfmt`/`shellcheck` ran against it.

**Acceptance Scenarios**:

1. **Given** a `.sh` file is edited through the Codex CLI adapter, **When** the edit completes, **Then** `shfmt -w -i 2` and `shellcheck` run against it, matching `.claude/hooks/post-edit-format.sh`'s current behavior.
2. **Given** a `.yaml`/`.yml` or `.json` file is edited, **When** the edit completes, **Then** `yamllint` or a JSON syntax check runs respectively, matching current Claude Code behavior.

### User Story 6 - Deduplicate the MCP catalog via a `@path` import (Priority: P2)

A maintainer wants `.claude/rules/mcp.md`'s MCP server catalog and usage rule (Q4) to be a single source of truth shared between Claude Code and Codex CLI, rather than two hand-maintained copies (`.claude/rules/mcp.md` and `AGENTS.md`) that can silently drift apart when one is updated and the other isn't. This directly serves this spec's stated purpose — the same configuration applied to both tools, not just parallel configurations built from shared logic.

**Why this priority**: Q4 is the cleanest dedup candidate in this feature: its content (an MCP server table and a usage rule) already carries no Claude Code-specific tool names or mechanism references, so nothing needs to diverge between the two files. It depends on User Story 1 (`AGENTS.md` must exist first) and touches a file (`.claude/rules/mcp.md`) outside this feature's original `.claude/hooks/`-focused footprint, so it ranks alongside the other P2 work rather than with the zero-risk P1 wins.

**Independent Test**: With `AGENTS.md`'s Q4 entry created (User Story 1), restructure `.claude/rules/mcp.md` to `@path`-import that entry instead of containing its own copy, and confirm a Claude Code session still sees the full MCP catalog and usage rule (now sourced from `AGENTS.md`) while a Codex CLI session sees the same text natively from `AGENTS.md` — independent of every other user story in this feature.

**Acceptance Scenarios**:

1. **Given** `AGENTS.md`'s Q4 entry (the MCP catalog table and usage rule) exists per User Story 1, **When** `.claude/rules/mcp.md` is restructured, **Then** it contains a `@path` import pointing at `AGENTS.md`'s Q4 content instead of a separately maintained copy of the same table.
2. **Given** a future update to the MCP catalog (e.g. a new server added), **When** the maintainer edits `AGENTS.md`'s Q4 entry, **Then** `.claude/rules/mcp.md`'s imported content reflects the change for Claude Code sessions automatically, with no second edit required.
3. **Given** Claude Code's `@path` import behavior for a file outside `.claude/` (a repository-root `AGENTS.md`) is unconfirmed by any source consulted in this feature, **When** this story is implemented, **Then** the import is verified against a live Claude Code session before being considered complete. If `@path` cannot target a file outside `.claude/`, `.claude/rules/mcp.md` MUST keep its current standalone content and the gap MUST be documented rather than left silently inconsistent with `AGENTS.md`.

### Edge Cases

- What happens if a destructive-command category's decision is "ask" but Codex CLI's hook contract only supports a binary allow/deny? The adapter MUST default to deny (fail-safe default), never allow.
- What happens if a shared script itself errors (crashes, times out, unexpected input)? Each adapter MUST treat a script error the same way it treats a "deny" decision, not an "allow" — a broken guardrail must fail closed, not fail open.
- What happens if `.claude/hooks/pre-bash.sh`'s behavior changes in the future (a new destructive-command category is added)? Because the matching logic is extracted into one shared script that both Claude Code's hook and Codex CLI's adapter call, a future addition made once in the shared script propagates to both automatically — this is a design property to preserve, not a scenario requiring new tests today.
- What happens if Codex CLI's actual hook behavior (once re-verified per FR-010/FR-022) turns out to differ from `research.md`'s R1 or from [learn.chatgpt.com/docs/hooks](https://learn.chatgpt.com/docs/hooks)'s claims? The Codex CLI adapter(s) affected MUST be adjusted to match the verified behavior before being considered complete; the acceptance scenarios that cite re-verification are blocking, not advisory.
- What happens to Claude Code's own `pre-bash.sh`, `pre-edit.sh`, and `post-edit-format.sh` behavior during their respective refactors? Each MUST produce identical outcomes before and after the shared-script extraction — this feature must not regress a guardrail that already works.
- What happens if Codex CLI's `PreToolUse`/`PostToolUse` file-edit coverage, once re-verified, turns out narrower than documented (e.g. covers `apply_patch` but not a raw `Write` tool call)? The Codex CLI adapters for User Stories 4–5 MUST be scoped to whatever is actually verified, and any gap MUST be documented rather than assumed closed.
- What happens if `AGENTS.md`'s note that "Codex CLI enforces this via hook" (Q7/Q9/Q10) is read by a tool other than Codex CLI? The entry MUST state the enforcement applies to Codex CLI specifically — it must not be phrased as if every reader of `AGENTS.md` gets the same protection, since only Codex CLI is a target of this feature.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The system MUST provide a root-level `AGENTS.md` file containing one entry for each of the 12 prose-appropriate items recorded in `specs/012-cross-agent-guardrail-migration/decision-record.md` (Q1, Q2, Q4, Q5, Q7–Q14). For Q4, this entry is the canonical source of that content — see FR-023, which restructures `.claude/rules/mcp.md` to import it rather than duplicate it.
- **FR-002**: For each item still purely prose (Q5, Q8, Q11, Q12, Q13), the `AGENTS.md` wording MUST NOT imply an enforcement guarantee — phrased as a note or request, per that item's decision-record rationale. For Q7, Q9, and Q10 — upgraded to real enforcement for Codex CLI per FR-016–FR-021 — the `AGENTS.md` entry MUST state plainly that Codex CLI enforces the item via hook.
- **FR-003**: Q2's `AGENTS.md` entry MUST use tool-agnostic vocabulary (no Claude Code-specific tool names).
- **FR-004**: `AGENTS.md` MUST NOT carry Q3's skill-routing table as a prose transcription — it is superseded by native `.agents/skills` sync (FR-013) for Codex CLI.
- **FR-005**: Q13's and Q14's `AGENTS.md` entries MUST NOT name a Claude Code-specific mechanism (Claude Memory, the `Agent` tool, `subagent_type` values).
- **FR-006**: The system MUST provide one shared script encoding `.claude/hooks/pre-bash.sh`'s destructive-command matching logic (force push, `git reset --hard`, `git clean -f`, `rm -rf` against root/home/cwd, other `rm -rf`, `mkfs`/`dd`/fork-bomb, `curl|bash`, non-HTTPS request, credential-path read, credential-path write, global package installs, `sudo`), exposing one tool-agnostic decision (allow / deny / ask) plus a human-readable reason per invocation.
- **FR-007**: The system MUST provide a Codex CLI adapter that invokes the shared script from a `PreToolUse` hook scoped to Bash tool calls and translates its decision into Codex CLI's expected hook response.
- **FR-008**: Where the shared script's decision is "ask" and Codex CLI's hook contract has no three-way primitive, the adapter MUST default to deny rather than allow.
- **FR-009**: `.claude/hooks/pre-bash.sh` MUST continue to produce identical allow/deny/ask outcomes for every existing test case after being refactored to call the shared script — zero behavior regression for Claude Code.
- **FR-010**: Before the Codex CLI destructive-command adapter is considered complete, the implementation MUST re-verify `research.md`'s R1 (Codex CLI `PreToolUse` scope and response shape) against Codex CLI's primary documentation, since the decision record flags it as Medium-confidence.
- **FR-011**: If `AGENTS.md` already exists at the time this feature is implemented, the system MUST surface its current content to the maintainer before any content is overwritten.
- **FR-011a**: `AGENTS.md`'s content MUST fit within Codex CLI's default 32 KiB size budget (`project_doc_max_bytes`, per [agent-configuration/agents-md](https://learn.chatgpt.com/docs/agent-configuration/agents-md)) — content beyond this budget is silently dropped by Codex CLI's discovery, so the 12-item entry set MUST be written concisely enough to stay under this limit, not assumed unbounded.
- **FR-012**: The system MUST provide an automated test suite covering every destructive-command category in FR-006, run against the shared script directly and against both integration points (Claude Code hook, Codex CLI adapter).
- **FR-013**: The system MUST make each of this repository's 9 custom-authored skills (`adr`, `advisor`, `clarifier`, `coder`, `domain-model`, `minto-builder`, `minto-reviewer`, `minto-rewriter`, `ubiquitous-language`) available under `.agents/skills/<name>/SKILL.md`, using the same directory-per-skill structure Codex CLI's documentation specifies, mirroring the sync mechanism already used for the 15 `speckit-*` skills.
- **FR-014**: `.agents/skills/<name>/SKILL.md` content for each of the 9 custom skills MUST remain equivalent to its `.claude/skills/<name>/SKILL.md` source — the two MUST NOT be allowed to silently drift apart after either is edited. Since both tools use the identical directory-per-skill, `SKILL.md`-required structure, the sync SHOULD be implemented as a filesystem symlink (`.agents/skills/<name>` → `.claude/skills/<name>`) rather than a copy — this makes the two paths the same file by construction, eliminating drift risk entirely rather than merely detecting it. A generated copy remains an acceptable fallback only if symlinks prove unworkable (e.g. a target environment that dereferences or forbids them).
- **FR-015**: This feature MUST NOT build or claim skill-file discoverability, or any other guardrail behavior, for Cursor or any tool other than Codex CLI — Cursor support is explicitly out of scope for this spec.
- **FR-016**: The system MUST provide one shared script encoding `.claude/hooks/pre-edit.sh`'s block logic (deny edits under `.git/`; deny edits when the current branch is `main`/`master`), exposing a tool-agnostic allow/deny decision plus reason, reused by both Claude Code's existing hook and a new Codex CLI adapter.
- **FR-017**: The system MUST provide a Codex CLI adapter that invokes FR-016's shared script from a `PreToolUse` hook scoped to `apply_patch`/`Edit`/`Write`, translating the decision into Codex CLI's expected response format.
- **FR-018**: `.claude/hooks/pre-edit.sh`'s block behavior (Q9, Q10) MUST continue to produce identical outcomes for every existing test case after being refactored to call the shared script — zero regression for Claude Code.
- **FR-019**: The system MUST provide one shared script encoding `.claude/hooks/post-edit-format.sh`'s per-extension formatting behavior (`shfmt`/`shellcheck` for `.sh`, `yamllint` for `.yaml`/`.yml`, JSON syntax check for `.json`), reused by both Claude Code's existing hook and a new Codex CLI adapter.
- **FR-020**: The system MUST provide a Codex CLI adapter that invokes FR-019's shared logic from a `PostToolUse` hook scoped to `apply_patch`/`Edit`/`Write`.
- **FR-021**: `.claude/hooks/post-edit-format.sh`'s behavior MUST continue to produce identical outcomes for every existing test case after being refactored to call the shared script — zero regression for Claude Code.
- **FR-022**: Before the Codex CLI pre-edit and post-edit adapters (User Stories 4–5) are considered complete, the implementation MUST re-verify [learn.chatgpt.com/docs/hooks](https://learn.chatgpt.com/docs/hooks)'s claims — specifically that `PreToolUse`/`PostToolUse` cover `apply_patch`/`Edit`/`Write` and that a deny decision blocks the edit before completion — against a live Codex CLI session, mirroring FR-010's re-verification requirement for Q6.
- **FR-023**: `.claude/rules/mcp.md`'s MCP server catalog and usage rule content (Q4) MUST be restructured to `@path`-import the corresponding `AGENTS.md` entry rather than maintain a separate copy, once `AGENTS.md` exists (FR-001) — eliminating duplication for this specific item.
- **FR-024**: Before FR-023 is considered complete, the implementation MUST verify that Claude Code's `@path` import syntax can target a file outside `.claude/` (a repository-root `AGENTS.md`) against a live Claude Code session. If it cannot, `.claude/rules/mcp.md` MUST keep its current standalone content, and the gap MUST be documented rather than left silently inconsistent with `AGENTS.md`.
- **FR-025**: `install.sh` MUST be extended to copy this repository's `AGENTS.md` to `~/.codex/AGENTS.md`, mirroring its existing `.claude/` → `~/.claude/` copy, so Codex CLI sessions in every project on the machine see the same guidance.
- **FR-026**: `install.sh` MUST be extended to make each of the 9 custom skills available under `~/.agents/skills/<name>/SKILL.md` (global scope, per Codex CLI's documented `$HOME/.agents/skills` discovery), consistent with FR-014's symlink-preferred approach.
- **FR-027**: `install.sh` MUST be extended to register the shared destructive-command, pre-edit, and post-edit scripts (FR-006, FR-016, FR-019) as Codex CLI hooks at global scope (`~/.codex/config.toml` `[hooks]` or `~/.codex/hooks.json`, per whichever this repository's Phase 0 research confirms is current), mirroring how `.claude/settings.json`'s hook wiring already applies once installed to `~/.claude/settings.json`.
- **FR-028**: `.claude/rules/mcp.md`'s `@path` import (FR-023) MUST target the installed global `AGENTS.md` location consistently with where `.claude/rules/mcp.md` itself resolves after installation (`~/.claude/rules/mcp.md`) — the import path MUST be verified to resolve correctly post-install, not only when read directly from this repository's working tree.

### Key Entities

- **AGENTS.md**: Shared, tool-agnostic instruction file at the repository root; the carrier of the 12 remaining prose-appropriate decision-record items (Q1, Q2, Q4, Q5, Q7–Q14; Q3's routing-table content is carried natively per FR-013 instead). For Q7, Q9, Q10, its entry additionally states that Codex CLI enforces the item via hook. Read natively by Codex CLI; any other reader is incidental, not a target of this feature.
- **Shared destructive-command script**: Single source of truth for `pre-bash.sh`'s matching logic (Q6), tool-agnostic in its inputs (a shell command string) and outputs (allow/deny/ask + reason).
- **Shared pre-edit script**: Single source of truth for `pre-edit.sh`'s block logic (Q9, Q10 — `.git/` paths, `main`/`master` branch), tool-agnostic in its inputs (tool name + file path) and outputs (allow/deny + reason).
- **Shared post-edit script**: Single source of truth for `post-edit-format.sh`'s per-extension formatting logic (Q7), tool-agnostic in its inputs (a file path).
- **Codex CLI `PreToolUse` adapter (Bash)**: Thin translation layer between the shared destructive-command script's decision and Codex CLI's `PreToolUse` hook contract, scoped to Bash.
- **Codex CLI `PreToolUse` adapter (edit)**: Thin translation layer between the shared pre-edit script's decision and Codex CLI's `PreToolUse` hook contract, scoped to `apply_patch`/`Edit`/`Write`.
- **Codex CLI `PostToolUse` adapter**: Thin translation layer between the shared post-edit script and Codex CLI's `PostToolUse` hook contract, scoped to `apply_patch`/`Edit`/`Write`.
- **`.agents/skills/` sync**: The mechanism that keeps this repository's 9 custom-authored skills discoverable by Codex CLI's native `.agents/skills` scan, without content drift from their `.claude/skills/` source — preferably a symlink per skill (FR-014) rather than a generated copy, unlike the existing `speckit-expand-update.sh` pattern for the `speckit-*` skills, which copies because it regenerates from an external CLI.
- **`.claude/rules/mcp.md` `@path` import**: The single point in this feature where a Claude Code-side file is restructured (not just a new Codex CLI-side artifact) to consume `AGENTS.md` as its canonical source for Q4, eliminating duplication for that one item (FR-023/024/028).
- **`install.sh` (extended)**: This repository's existing global installer, extended to also deploy the Codex CLI-side artifacts this feature produces (`AGENTS.md`, the 9 skill symlinks, the three shared guardrail scripts' hook registrations) to their global locations, matching its existing `.claude/` → `~/.claude/` behavior (FR-025–028).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: All 12 remaining prose-appropriate decision-record items (Q1, Q2, Q4, Q5, Q7–Q14) have a corresponding, verifiable entry in `AGENTS.md` (12/12, checked by a reviewer against `decision-record.md`); Q3 and Q6 are verified instead by SC-006 and SC-002 respectively.
- **SC-002**: Every destructive-command test case that currently passes against `.claude/hooks/pre-bash.sh` produces the identical allow/deny/ask outcome when run through the Codex CLI adapter (100% parity between Claude Code and Codex CLI).
- **SC-003**: `.claude/hooks/pre-bash.sh`'s own existing behavior shows zero regressions after the shared-script refactor (100% of its current test cases still pass).
- **SC-004**: A maintainer working from a fresh Codex CLI session in any installed project (no prior context of spec 012) can, by reading only the installed `AGENTS.md` (`~/.codex/AGENTS.md`), correctly state for any of the 12 items whether it is enforced or advisory-only.
- **SC-005**: Attempting a hard-blocked destructive command from a Codex CLI session in any project on a machine that has run `install.sh` is denied before execution, with no successful execution path found during testing.
- **SC-006**: All 9 custom-authored skills exist under `.agents/skills/<name>/SKILL.md` with content matching their `.claude/skills/<name>/SKILL.md` source (9/9, verified by diff), and a representative request routed through a Codex CLI session triggers the correct skill.
- **SC-007**: Attempting to edit a file under `.git/`, or any file while on the `main`/`master` branch, from a Codex CLI session in any installed project is denied before the edit completes, with no successful edit path found during testing (Q9, Q10).
- **SC-008**: Editing a `.sh`, `.yaml`/`.yml`, or `.json` file from a Codex CLI session in any installed project results in the same formatting/lint pass Claude Code already applies, verified for each extension category (Q7).
- **SC-009**: `.claude/hooks/pre-edit.sh` and `.claude/hooks/post-edit-format.sh` show zero regressions after their respective shared-script refactors (100% of their current test cases still pass).
- **SC-010**: `.claude/rules/mcp.md` contains no standalone copy of the MCP catalog/usage-rule text after this feature ships — only a `@path` import of `AGENTS.md`'s Q4 entry (or, if FR-024's verification fails, a documented explanation of why the import wasn't possible).
- **SC-011**: Running `install.sh` on a machine with neither `~/.codex/` nor `~/.agents/` yet deploys `AGENTS.md`, the 9 skills, and the three Codex CLI hook registrations correctly on the first run (no manual follow-up steps), and re-running `install.sh` afterward is idempotent (no duplicated hook entries, no broken symlinks).

## Assumptions

- Codex CLI is actively used to work in this repository going forward — the premise of the original request.
- **Cursor support is explicitly out of scope for this spec**, per the maintainer's explicit instruction. No Cursor-targeting code, `AGENTS.md` wording, or test is produced by this feature. A future spec may extend Cursor coverage using the same shared-script pattern established here, since each shared script's inputs/outputs are already tool-agnostic.
- `specs/012-cross-agent-guardrail-migration/research.md`'s R1 (Medium confidence) must be re-verified against Codex CLI's primary documentation before the Codex CLI destructive-command adapter ships (FR-010). R4's premise (no tool has a pre-edit file-write block) is already superseded for Codex CLI by Follow-up correction 2. R5 (AGENTS.md read natively) is already High confidence and needs no re-verification. R2/R3 (Cursor-specific) are not load-bearing for this feature.
- No test suite currently exists for `.claude/hooks/pre-bash.sh`, `.claude/hooks/pre-edit.sh`, or `.claude/hooks/post-edit-format.sh` (confirmed: `tests/` has no suite for any of the three); suites for all three must be created as part of this feature.
- `AGENTS.md` is the file Codex CLI reads for general instructions, distinct from `.agents/skills/` (skills) — this feature does not introduce a separate `CODEX.md`.
- Where a shared script returns "ask" and Codex CLI's hook contract has no three-way decision primitive, the fail-safe default (deny) applies, consistent with this repository's existing least-privilege / fail-safe-default permission philosophy.
- Codex CLI's `.agents/skills` discovery mechanism ([learn.chatgpt.com/docs/build-skills](https://learn.chatgpt.com/docs/build-skills), corroborated by [agent-configuration overview](https://learn.chatgpt.com/docs/customization/overview)) and its `PreToolUse`/`PostToolUse` file-edit coverage ([learn.chatgpt.com/docs/hooks](https://learn.chatgpt.com/docs/hooks)) are treated as sufficiently confirmed for planning (documented, named sources, consistent with this repository's own existing `.agents/skills/speckit-*` usage and with `.claude/hooks/pre-edit.sh`'s/`post-edit-format.sh`'s own contract shape) but must still be spot-checked against a live Codex CLI session before the relevant user stories are considered complete (FR-022, and an equivalent spot-check for User Story 2).
- `AGENTS.md`'s discovery hierarchy (global `~/.codex/AGENTS.md`, then project-scope root→cwd with `AGENTS.override.md` taking precedence per directory, per [agent-configuration/agents-md](https://learn.chatgpt.com/docs/agent-configuration/agents-md)) means a single root-level `AGENTS.md` is sufficient for this repository's current needs — no nested per-directory `AGENTS.md` files or an `AGENTS.override.md` are required, since this repository has no directory-specific guidance that should override the root file.
- Codex CLI's "Memories" feature (`~/.codex/memories/`, per [customization/memories](https://learn.chatgpt.com/docs/customization/memories?surface=app)) is user-scoped and auto-generated from session history — the documentation itself recommends keeping "required team guidance in AGENTS.md" rather than relying on it. This confirms, rather than changes, Q13's original decision-record treatment (mechanism-agnostic prose in `AGENTS.md`, no attempt to integrate with Codex's Memories feature).
- Codex CLI's "Rules" feature (Starlark `.rules` files under `~/.codex/rules/` or `<repo>/.codex/rules/`, `prefix_rule()` with `allow`/`prompt`/`forbidden` decisions, per [agent-configuration/rules](https://learn.chatgpt.com/docs/agent-configuration/rules)) is a more native command-permission mechanism than this spec's planned `PreToolUse` shell-script adapter (FR-006/007) for Q6, but its literal-prefix/union matching appears less expressive than `pre-bash.sh`'s embedded-regex matching. Whether to use Rules for the subset of categories it can express (e.g. `sudo`, exact global-install subcommand prefixes), the `PreToolUse` adapter for the rest, or the adapter alone, is deferred to `/speckit-plan` — not decided by this spec.
- Codex CLI's "Subagents" feature (TOML files under `~/.codex/agents/` or `.codex/agents/`, per [agent-configuration/subagents](https://learn.chatgpt.com/docs/agent-configuration/subagents?surface=app)) is a real mechanism beyond what Q14's recorded verdict (mechanism-agnostic prose) requires. Defining actual Codex subagent configs for this repository is a plausible future enhancement but is explicitly **not** part of this spec's scope — FR-001–FR-022 do not include it, and adding it would require a new spec or an explicitly scoped addition, not an assumption.
- Claude Code does not read `AGENTS.md` (confirmed by [features-overview](https://code.claude.com/docs/en/features-overview) and [claude-directory](https://code.claude.com/docs/en/claude-directory) — it reads `CLAUDE.md` and `.claude/rules/*.md` exclusively). True single-file sharing of prose content is therefore only possible via Claude Code's `@path` import pointing *at* `AGENTS.md`, not via a file both tools read natively — this is why FR-023's dedup is a one-directional import (Claude Code imports from Codex CLI's native file), not a symmetric shared file.
- Q1 (`pre-edit.sh` CI/settings/production warnings) is a **secondary, deferred** dedup candidate: its content is tool-agnostic like Q4's, but no existing `.claude/rules/*.md` file currently holds it as standalone prose to redirect via `@path` — the warning text lives only inside the hook script's own output today. Extending FR-023's pattern to Q1 would require first extracting that text into a rules file, which this spec does not undertake; `/speckit-plan` may propose it as a follow-on if FR-023's approach proves out for Q4.
- Q2, Q13, and Q14 are **not** dedup candidates: their `AGENTS.md` wording is deliberately different from `.claude/rules/tools.md`'s own content (genericized per FR-003/FR-005), so a `@path` import would either import the wrong (Claude Code-specific) text or require maintaining two divergent files under one import — neither serves this feature's goal. Q5, Q8, Q11, and Q12's current `.claude/rules/permissions.md` footprint mixes Claude Code-specific hook-mechanism description with the portable policy statement; separating them cleanly enough to dedup is a larger rewrite than this feature undertakes.
- Per the Clarifications session above, "align `.claude/` with official docs" is scoped to FR-023 only. `.claude/agents/*.md` (real Claude Code subagent definitions) remains absent by choice, not oversight: adding it here without a matching Codex CLI-side subagent definition (already deferred, see the Subagents assumption above) would create a new asymmetry rather than resolve one, so it is left for a future spec if both sides are ever built together.
- Per the second Clarifications entry, this repository's own `README.md` ("designed to be copied wholesale to `~/.claude/`... apply across every project on the machine") and `.claude/settings.json`'s already-global hook paths (`"$HOME"/.claude/hooks/...`) establish that this repository is a distributable global configuration, not a self-contained project. This feature's Codex CLI artifacts (`AGENTS.md`, `.agents/skills/`, the shared guardrail scripts) follow the same model: authored here, deployed globally by `install.sh` (FR-025–028). Every acceptance scenario and success criterion phrased around "a Codex CLI session" therefore means *any* project on a machine that has run `install.sh`, not only sessions working on this repository itself.
- `install.sh`'s exact current mechanics (how it copies `.claude/`, registers MCP servers) are treated as a known starting point for planning but MUST be read in full during Phase 0 research before FR-025–028 are implemented, so the extension follows its existing idempotency and error-handling conventions rather than introducing a divergent style.
