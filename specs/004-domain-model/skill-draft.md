# Skill: domain-model — 下書き

> ステータス: 要件定義中。実装前にこの内容を spec.md へ昇格させること。

## 概要

プロジェクトのドメインモデルを維持する — DDD の構成要素（集約・エンティティ・値オブジェクト・ドメインイベント・不変条件）を会話から収集し、`docs/models/<context>.md` に整理し、`docs/models/index.md` でインデックス管理する。

## 確定した要件

| # | 要件 | 決定内容 |
|---|------|---------|
| R1 | 成果物形式 | DDD構成要素のMarkdownテーブル + Mermaid クラス図 |
| R2 | UL連携 | `docs/ubiquitous-language.md` が存在すれば読み込んで候補を提案、なければ独立ブートストラップ |
| R3 | トリガー方式 | 明示コマンド + UL同様の受動監視（会話中のDDD語彙をキューに積む） |
| R4 | 出力言語 | 日本語 |
| R5 | 複数モデル設計 | 1コンテキスト = 1ファイル（`docs/models/<context-kebab>.md`）、インデックスは `docs/models/index.md` |

## 受動収集トリガー（案）

| パターン | 例 | trigger_type |
|---|---|---|
| 「〇〇は〇〇を持つ」「〇〇に属する」 | 「注文は複数の明細を持つ」 | `aggregate` |
| 「〇〇ID」「〇〇番号」「〇〇コード」 | 「注文IDで一意に識別される」 | `entity` |
| 「変更されない」「置き換えられる」「同値なら同一」 | 「住所は変更されない」 | `value-object` |
| 動詞+名詞の過去形/受動形 | 「注文が確定された」 | `domain-event` |
| 「〜の場合は〜できない」「〜でなければならない」 | 「在庫がゼロの場合は注文できない」 | `invariant` |

## 出力ファイル構成（案）

```
docs/models/
  index.md                      # BC一覧 + コンテキスト間関係図
  <bounded-context-kebab>.md    # BC別モデルファイル（1 BC = 1 file）
```

### index.md の構成

- プロジェクト名・更新日
- Bounded Context 一覧（リンク付き）
- Mermaid によるコンテキスト間関係図（任意）

### <context>.md の構成

1. Mermaid classDiagram（テーブルから機械生成）
2. 集約テーブル（集約名・集約ルート・責務・境界・不変条件・ライフサイクル・実装名）
3. エンティティテーブル（エンティティ名・識別子・属性・振る舞い・状態遷移・所属集約・実装名）
4. 値オブジェクトテーブル（VO名・構成要素・検証ルール・不変性理由・使用箇所・実装名）
5. ドメインイベントテーブル（イベント名・トリガー・ペイロード・後続処理・実装名）
6. 不変条件リスト（条件 / 制約 / 違反時の扱い）

## 不変条件（案）

1. **Diff before write**: 書き込み前に必ず差分を提示する
2. **Explicit confirmation**: 明示確認なしにファイルを書き込まない
3. **No silent cross-context merge**: コンテキスト間の概念衝突は必ず分離か改名かを提示する
4. **Single-file per BC**: 1コンテキスト = 1ファイル、混在不可
5. **Index always reflects files**: `index.md` は全コンテキストファイルを常に同期
6. **Diagram from tables**: Mermaid図はテーブルから機械生成、乖離時は再生成を提案

## 未決事項（要検討）

- 集約テーブルのフィールド数と粒度（7フィールドにするか、より少なくするか）
- Bootstrap時のElicitation順序（集約→エンティティ/VO→イベント が妥当か）
- ULスキルとの重複管理（ドメインイベントはULにも記録される。どちらが正とするか）
- コンテキスト間関係図の形式（Mermaid graph か context-map.md のテーブル形式か）
